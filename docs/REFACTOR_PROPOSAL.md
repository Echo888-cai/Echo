# Echo Research 商业化重构提案 v2（2026-07-12）

> **这是什么**：Echo Research 从"邀请制 beta 原型"走向"可长期演进的商业化产品"的整体技术方案。
> **前提设定**（决策者定调）：不设成本约束；以**稳定性、技术美感、未来价值最大化**为三大最高准则。
> **地位**：独立提案，供两位创始人共同评审。不改动 docs/PLAN.md（宪法），通过后再并入主计划。
> **审查依据**：2026-07-12 对全部代码分层（routes/services/repositories/ui）、21 个数据库迁移、数据接口层（market/marketData/financialData/fmpClient/secFilings）、部署资产与 PLAN v1–v5 的完整审读。结论承接项目审计传统：以事实为证，不凭印象。

## 0. 一句话论点

Echo Research 的价值不在框架，而在**领域核心**（估值交叉验证、factGuard 防幻觉、三市一手财报管道、证伪闭环）与**品牌气质**（安静的研究札记）。重构的使命是：把这两样资产从"单机原型的底盘"上完整取下，安放到一个**十年不用换血的底盘**上——每一次 10 倍增长（10 人 → 千人 → 十万人 → 更多）都只做加法，不推倒重来。方法是绞杀者模式：任何时刻主干可发布，领域逻辑只搬家不重写，每一步有回滚线。

## 1. 三大准则的工程化定义

抽象的价值观必须翻译成可执行、可验收的设计原则，否则会沦为口号。

### 1.1 稳定性 → 可靠性工程（Reliability Engineering）

- **SLO 先行**：上线前定义服务目标并持续度量——API 可用性 99.9%、研究问答 P95 首 token < 3s、数据管道日更成功率 99%、RPO ≤ 5 分钟、RTO ≤ 30 分钟。没有度量的稳定只是运气。
- **失败是常态设计**：现有代码已有优秀的降级基因（任一数据源超时返回 missing 占位、模型多级 fallback、诚实说"取不到"）。重构把这条基因升格为全局契约：**每一层都必须声明自己的降级形态**，并在混沌演练中被真实验证。
- **状态收敛**：稳定性最大的敌人是散落的状态。目标架构里，Web 层完全无状态、任务全部进队列、唯一的状态在托管数据库与对象存储——挂掉任何一台计算节点，用户无感知。

### 1.2 技术美感 → 结构的必然性

美感不是用最新的技术，而是**每个组件的存在都有不可辩驳的理由，且边界清晰到可以独立解释**：

- **六边形架构（端口与适配器）**：领域核心（估值、财务质量、证伪、factGuard）是**纯函数库**——不 import 数据库、不 import HTTP、不 import 任何框架。外界通过端口（接口）注入数据，核心输出判断。这让估值引擎可以在服务端、worker、测试、甚至未来的边缘环境中原样运行，也让"换框架"永远伤不到灵魂。
- **契约即真理**：前后端、服务与服务之间的一切通信由单一 schema 源（zod）定义，编译期生成类型与 OpenAPI 文档。契约变更 = 代码评审可见的 diff，杜绝"口头对齐"。
- **金融数值的精确性**：JS 浮点数不配承载金融计算。所有金额、股数、比率在存储与计算层使用 decimal 语义（数据库 NUMERIC + 应用层 decimal 库），浮点只允许出现在最终展示层。这是金融产品的教养。
- **决策留痕（ADR）**：每个重大技术决策写一页 Architecture Decision Record（背景 → 选项 → 决定 → 代价）。项目已有 PLAN.md 的"红线 + 触发条件"传统，ADR 是它的工程化延续——十年后接手的人能读懂每一块石头为什么放在那里。

### 1.3 未来价值最大化 → 资产化

把"将来可能值钱的东西"从代码里识别出来，按资产而非功能来建设：

| 资产 | 现状 | 资产化方向 |
|---|---|---|
| 一手财报数据管道（CNINFO/HKEX/EDGAR） | 内嵌在应用服务里 | 升格为独立**数据平台**，双时态（bitemporal）存储，成为未来回测、量化研究、B 端数据服务的地基 |
| 研究判断与证伪记录 | 散落在 sessions/snapshots 表 | **append-only 事件日志**：每个判断、每次证伪核对、每条 factGuard 审计不可变追加——既是审计合规资产，也是未来训练专有模型的语料 |
| factGuard + 评测体系 | shadow 模式审计 | 升格为 **AI 质量平台**：黄金问题集回归评测、模型路由预算、语义缓存——换模型从"手感"变成"跑分" |
| 品牌视觉（牛皮纸/墨色/陶土/静谧动效） | 分层 CSS | 独立**设计系统包**（design tokens + 组件库），Web/PWA/App/营销站/分享图共享同一套气质 |
| API 面 | 内部使用 | OpenAPI 契约从第一天按"未来开放平台"的标准书写——版本化、幂等、错误码规范 |

## 2. 现状审查结论（保留什么，替换什么）

### 2.1 必须原样保留的资产（重写 = 毁灭价值）

- `src/server/services/` 全部领域逻辑：valuationEngine、financialQuality、factGuard、falsifyRules、answerComposer、eventEngine、companyPortrait、insiderActivity、historicalValuation、earningsCalendar。
- 三市数据管道：cnFilingsPipeline（CNINFO 91% 覆盖）、hkFilingsPipeline（HKEX PDF 表格抽取）、secFilings（EDGAR + Form 4 真实解析）。
- 工程纪律：编号化迁移、每阶段测试文件、红线制度、"诚实降级"产品哲学。
- 品牌与交互范式：单一研究对话流、证据溯源卡、数据接地条、置信度体系。

### 2.2 需要更换的底盘（当前形态的天花板）

| 组件 | 现状 | 天花板 |
|---|---|---|
| 运行时结构 | 单进程：HTTP + SSE + scheduler + PDF 解析同灶 | 一个卡顿的 PDF 解析影响所有在线用户；横向扩容时 scheduler 重复执行 |
| 数据库 | SQLite（WAL） | 单机文件锁；无 RLS；JSON-in-TEXT 列（company_details 十余个 JSON 字段、sessions 的 thread_json）无约束不可查询 |
| HTTP 层 | 手写 if/startsWith 路由，鉴权/CSRF/限速内联 | 无中间件体系、无输入校验框架、无 OpenAPI，多端接入靠人肉对齐 |
| 前端 | 无框架原生 ESM，20 个文件靠约定协作 | 无组件复用体系、无类型化契约、无法支撑多端与高速体验迭代 |
| 类型系统 | JS + jsconfig checkJs | 服务间契约靠注释；团队扩大后必然互踩 |
| 可观测性 | llm_audit / fact_guard_audit（优秀雏形）+ console | 无 trace、无指标告警，回答不了"用户经历了什么" |

## 4. 技术选型（决定 + 理由 + 被否决的替代项）

### 4.1 语言：TypeScript 全栈统一

- **理由**：现有代码是 JS ESM + jsconfig 检查，是全行业迁移成本最低的路径；契约层要求前后端共享 schema，同语言是最优雅的实现；领域核心以纯函数库形态存在时，TS 的类型系统足以表达其全部约束。
- **被否决**：Java/Go 重写后端——领域逻辑移植成本高且引入双语言栈的永久摩擦；**保留的口子**：数据平面的高吞吐抓取 worker 若实测成为瓶颈，可用 Go/Rust 实现单个 worker（队列协议天然跨语言），无需全局换语言。
- **精度纪律**：全库金额/比率使用 Postgres NUMERIC；应用层引入 decimal 库；ESLint 自定义规则禁止领域核心出现裸浮点运算。

### 4.2 后端：NestJS 模块化单体 + BullMQ worker

- **理由**：现有 routes/services/repositories 与 NestJS 的 controller/service/repository 一比一对应，迁移是搬家不是重写；依赖注入让六边形架构的端口注入自然落地；自带 OpenAPI 生成、守卫（鉴权）、拦截器（审计）、管道（校验）。
- **单体而非微服务**：模块边界清晰的单体在可预见的规模内是**更稳定**的选择（少一个网络就少一类故障）。微服务的触发条件记录在案：单模块团队 ≥3 人独立发版需求 / 单模块资源画像显著异化。达到前不拆。
- **worker 分离是第一优先**：scheduler 全部任务迁入 BullMQ 队列（重试、退避、死信、并发控制、按用户分组），worker 独立部署。这是稳定性收益最大的单项改造。

### 4.3 数据库：PostgreSQL + Drizzle ORM

- **PostgreSQL 16+，托管多可用区**（AWS RDS / GCP Cloud SQL / 阿里云 RDS，按最终部署区域定）：自动故障转移、PITR 到任意秒级时间点、只读副本随需。
- **多租户双保险**：应用层显式 user_id 过滤（沿用现有红线 18 的 code review 硬检查）+ 数据库**行级安全（RLS）**兜底——即使应用层出 bug，越权查询在数据库层被拒绝。这是"稳定性"在数据安全维度的体现。
- **JSON 列结构化**：company_details 的 summary/moat/risks 等、sessions 的 decision_panel 拆为结构化表；确属半结构化的（模型原始输出）用 JSONB + CHECK 约束，可索引可查询。
- **双时态财务仓库**（数据平面核心 schema）：每条财务事实带两个时间轴——`valid_time`（属于哪个报告期）与 `knowledge_time`（何时被知晓/修正）。财报重述不覆盖旧值而是追加新知识版本。**这是未来做回测与量化研究的先决条件**（point-in-time 正确性），也是市面上绝大多数数据产品做不到的差异化。
- **Drizzle 而非 Prisma**：贴近 SQL、迁移文件是可审查的纯 SQL——延续现有"编号迁移、非破坏性、可回滚"的纪律。
- **pgvector**：研报与证据语义检索的将来时，schema 预留。
- **被否决**：ClickHouse/TimescaleDB 现阶段——产品是研究型而非 tick 级交易终端，日线粒度 Postgres 分区表绰绰有余；触发条件（分钟级行情历史 / 分析型查询拖慢主库）记录在案。

### 4.4 前端：React 19 + Vite + TypeScript

- **组织**：TanStack Query（服务端状态）+ TanStack Router（类型化路由）+ zustand（少量本地状态）；SSE 流式沿用现有交互范式。
- **设计系统包 @echo/ui**：以现有 00-foundation → 07-brand 分层 CSS 为蓝本提炼 design tokens（色彩/字阶/间距/动效曲线），Radix 无头组件承载交互可达性，**像素级保留品牌气质**。设计系统独立发包、独立 Storybook 文档——品牌资产的工程形态。
- **不用 Next.js**：登录后应用无 SEO 诉求，SPA + CDN 是更简单稳定的形态；营销官网独立静态站（可用 Astro），与应用解耦。
- **移动端**：PWA 先行（离线壳 + 推送）；原生体验诉求验证后上 Expo/React Native，复用契约与设计 token。

### 4.5 数据接口层：供应商适配器矩阵

现有 market.js 的"市场识别 + 每源 symbol 拼写"思想是对的，升格为正式的**适配器矩阵**：

- 每个数据供应商一个适配器，实现统一端口（Quote/Fundamentals/News/Filings/Calendar），并携带**授权元数据**（商用许可状态、配额、SLA、成本）——路由器按"授权允许 → 数据质量 → 延迟"三级排序选源，未授权源在商用环境自动不可选。**合规从人为纪律变成类型系统约束**。
- 商用数据源按"不设成本约束"原则直接对接机构级：A 股（Wind / 恒生聚源 / 东方财富 Choice 任一）、港股（HKEX 授权分销商）、美股（Polygon / Intrinio / FactSet 按需）。自有一手管道（CNINFO/HKEX/EDGAR）继续作为差异化层与交叉验证源。
- 所有外部响应进入数据平面前过**数据质量守卫**（factGuard 思想前移到入库环节：量纲、币种、可比性、异常跳变检查），带质量评分入库。

### 4.6 AI 平台

- modelGateway 升格为独立模块：多供应商路由（现有 fallback 链保留）、每用户/每功能预算、语义缓存（相同公司相同意图短时间窗命中缓存，稳定性与成本双赢）。
- **黄金评测集**：从真实会话沉淀 100–300 个带标准答案要素的问题，每次换模型/改 prompt 跑回归评分（事实准确率、结构完整度、factGuard 命中率）。llm_audit + fact_guard_audit 的既有积累直接成为评测基建。
- factGuard 按既定路线用真实误报率数据决定 shadow → soft → full 升档，节奏不变。

### 4.7 基础设施与稳定性工程

- **云与区域**：主流云托管服务（香港/新加坡区起步，兼顾巨潮与腾讯源可达性 + 面向全球华人；境内合规主体确立后再评估境内区）。
- **环境**：production / staging 全等（同 IaC 模板拉起）+ 每 PR 预览环境。一切基础设施 Terraform 化——环境是代码，不是某台服务器上的手工记忆。
- **发布**：蓝绿部署 + 数据库迁移与代码发布解耦（先扩展后收缩的 expand-contract 迁移法）；任何发布可在 5 分钟内回滚。
- **备份**：Postgres PITR + 每日快照异地双份 + **每月恢复演练**（备份的定义是"演练过恢复的备份"，延续现有"校验备份本身"的正确直觉）。
- **可观测**：OpenTelemetry 全链路（一次研究问答 = 一条 trace：意图分类 → 数据采集各源耗时 → 模型调用 → factGuard → 落库），接 Grafana Cloud / Datadog；SLO 仪表盘 + 错误预算告警。
- **安全**：秘密管理进云 KMS（.env 退役）、依赖漏洞扫描进 CI、商用前一次第三方渗透测试、全量审计日志（谁在何时读了哪个用户的什么数据）。

## 5. 改造路线（绞杀者模式，六阶段）

> 不变式：**任何时刻主干可发布**；领域逻辑只搬家不重写；每阶段有独立验收与回滚线。现有 beta 在旧底盘上继续运行并产生反馈，直至 R-4 切流完成。

| 阶段 | 内容 | 验收标准 | 回滚线 | 状态 |
|---|---|---|---|---|
| **R-0 · 契约固化**（≈2 周） | 全部 /api/* 端点以 zod+OpenAPI 描述；现有 phase 测试改造为契约测试基线；黄金评测集 v1 落地 | 契约测试覆盖全部端点并在旧实现上全绿 | 纯增量，无回滚问题 | ✅ `packages/contracts` 落地（17 个 schema 模块），契约测试雏形在 `contract-tests/` |
| **R-1 · 数据底盘**（≈4 周） | Monorepo 骨架；Postgres schema 设计（结构化 + 双时态财务仓库 + RLS）；Drizzle 重写 repositories；SQLite→Postgres ETL + 新旧双跑对比 | 新旧库各跑契约测试输出等价；ETL 幂等可重放 | 保留 SQLite 路径直至 R-4，随时切回 | ✅ `packages/db` Drizzle schema + `sqlite-to-postgres.ts` ETL 落地；**生产仍在 SQLite**，Postgres 切流未发生 |
| **R-2 · 应用层平移**（≈4 周） | services 逐模块迁入 NestJS；scheduler 拆 BullMQ worker；鉴权升级正式 session/OIDC（预留微信/手机号登录）；OTel 埋点随迁移落地 | 每模块切流前后契约测试等价；worker 故障注入演练通过 | Caddy 按路由灰度，任一模块可切回旧 server.js | ✅ `apps/api`（17 模块）+ `apps/worker`（BullMQ）落地；**生产仍在 server.js**，OIDC/微信登录未做 |
| **R-3 · 体验层重建**（≈6 周，与 R-2 并行) | 设计系统包提炼；React 应用重建；类型化 client 自动生成；PWA 壳 | 品牌视觉评审通过（对照现有 UI 逐页走查）；核心链路（研究→关注→持仓→通知）真实账号全通 | 新旧前端并存于不同路径，可整体切回 | ✅ `packages/ui` token 提取 + `apps/web`（5 片：外壳/持仓/看盘/研究对话/设置）落地；PWA 壳未做；**生产仍在 src/app.js** |
| **R-4 · 数据平面独立**（≈4 周） | 供应商适配器矩阵 + 授权元数据路由；商用数据源接入替换腾讯/新浪；数据质量守卫入库检查；一手管道迁入 worker | 授权矩阵在商用配置下无未授权源可选；数据质量评分覆盖全部入库路径 | 适配器逐源切换，任一源可独立回退 | 🟡 适配器矩阵四端口 + 质量守卫已落地（ADR-0001）；一手管道已物理迁入 `apps/worker`（ADR-0002）；**商用数据源接入未做**，卡在 D6 询价签约 |
| **R-5 · 商业化硬化**（≈3 周） | 订阅计费（微信/支付宝支付对接）；审计日志全量；渗透测试与修复；SLO 仪表盘上线；恢复演练制度化 | 渗透测试报告闭环；SLO 连续两周达标；一次完整灾难恢复演练成功 | 计费模块独立开关 | ⬜ 未启动 |

总周期约 **5–6 个月**。R-0～R-3 的新底盘骨架均已按"搬家不重写"原则落地（`apps/*` + `packages/*`），但**生产流量至今仍完全跑在旧底盘**（`server.js` + `src/` + SQLite）——这是设计使然，不是延误：绞杀者模式的不变式是"任何时刻主干可发布"，切流是一个独立的、需要显式验收的决定，不会因为新底盘代码写完就自动发生。R-4 的适配器矩阵与一手管道搬迁已完成，商用数据源接入仍卡在 §7 D6（需询价签约，产品/商业决策，非技术缺口）。R-5 完全未启动，是当前路线图上唯一一个"零基础"的阶段——订阅计费、渗透测试、SLO 仪表盘、恢复演练四项都还没有代码或流程雏形，且大多数依赖前置闸门（§6 的行情数据商用授权、法务意见）先有结论。

## 6. 商业化前置闸门（与技术并行启动，缺一不可）

1. **行情数据商用授权**：询价与签约流程立即启动（A 股机构级数据商周期以月计）。授权状态进入适配器元数据，由类型系统而非纪律保证合规。
2. **业务资质边界法务意见**：产品定位（研究工具/信息服务，不输出个股买卖指令）与《证券法》证券投资咨询业务边界的书面法务意见；主体架构建议（境内运营 vs 香港主体先行）。
3. **生成式 AI 与数据合规**：目标市场若含中国大陆——算法备案/生成式 AI 备案、ICP、等保、个保法数据出境评估；若香港/海外先行——PDPO/GDPR 相应义务。此项直接决定部署区域拓扑，需在 R-1 的 schema 设计前有初步结论（数据驻留字段预留）。
4. **单用户成本模型**：llm_audit 既有数据推算每活跃用户月成本（模型+数据+搜索），输出定价模型草案。不设成本约束指的是基建投入，不是对单位经济学失明——**未来价值最大化的前提是这门生意的单位经济学成立**。

## 8. 结语：什么是这版方案里的"美感"

不是新技术的堆叠，而是四个结构性的干净：

1. **领域核心零依赖**——估值与判断的逻辑十年后仍能在任何运行时原样执行；
2. **契约单一源**——所有端之间没有口头协议，只有可 diff 的 schema；
3. **数据资产双时态**——系统永远记得"当时知道什么"，这让每个历史判断可复盘、可回测、可自证；
4. **失败有形态**——每一层的降级行为是声明式的、被演练过的，而不是祈祷。

这四点分别对应：未来价值、协作效率、金融产品的诚实、以及稳定性的本质。它们与 Echo Research 从第一天就坚持的产品哲学——诚实的置信度、可证伪的判断、安静的界面——是同一种气质在工程层的表达。
