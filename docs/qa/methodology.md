# 智能分析测评方法论

这份文档是"怎么测"的底账。语料和脚本会变，这里的原则不该变——每次复跑前先读一遍，
避免把测评本身退化成"跑一遍绿灯截图"。

## 为什么要有这套东西

Echo 的头号缺陷类型是 PLAN §7 记的**冻结表模式**：写好了没人调，不报错、不告警、只是永远不生效。
`check:frozen` 挡住了数据库表这一种形态，但同一个病还有别的形态：

- **死请求字段**：契约里有、前端发了、后端零读取（2026-07 抓到 `compareWith` / `documents`）。
- **死防线**：领域层算对了，在提示词/渲染边界被丢掉（抓到同业倍数上限）。
- **同义反复**：函数返回了一个"数字"，但它只是输入的常数变换（抓到 `base == 现价`）。

这三类的共同点：**单元测试全绿、类型检查全绿、CI 全绿，功能就是不存在**。
只有"拿真实问题跑真实链路、然后读输出"才抓得到。这就是这套测评的存在理由。

## 四条不许妥协的原则

### 1. 跑真函数，不跑副本

静态层 import 的是生产代码本体（`@echo/domain` 的意图分类、`apps/web/src/lib/resolve.ts`
的解析路由）。**绝不允许**为了好测而在测试里抄一份正则——副本会在源函数改了之后继续绿着骗人，
那正是我们要抓的病，不能由测评自己制造。

### 2. 期望写"应该怎样"，不写"现在怎样"

`corpus.ts` 每条 `expect` 是产品判断，不是当前行为的快照。快照测试会把既有缺陷固化成"预期行为"。

**跑挂了不许直接改期望。** 必须显式裁决一次：是代码错，还是期望错？
两者都可能（2026-07 首跑 40 条失败里就有 3 条是我的期望写错了）。裁决完，把理由写进 `why`。

### 3. live 层不 mock 任何外部源

mock 出来的绿灯正是冻结表模式的温床。live 抽样直接打 `runAsk`，走真实 DB、真实行情源、
真实 FMP/HKEX filing、真实 Tavily、真实模型。跑得慢（约 3 分钟）是这条路的成本，不是缺陷。

抽样量刻意小（15 条）且**串行**：并发会把供应商限流的锅算到被测代码头上。

### 4. 测不到就说测不到，不要靠推理补

2026-07 那次，浏览器 pane 的 `visibilityState` 恒为 `hidden`、视口报 `0×0`，
rAF 掉帧率**测不出来**。正确做法是在报告里写"未能验证"，而不是从 CSS 推理出一个
听起来合理的卡顿结论。**一个诚实的"不知道"比一个自信的猜测有用得多。**

同理，量到了但量级很小的东西要如实说小。那次测出流式渲染有 24× 的浪费，
但摊到 20 秒里只有 68ms——报成"卡顿元凶"就是不诚实。写清楚"真实存在但当前不是瓶颈"。

## 分层：每层抓什么

| 层 | 手段 | 抓什么 | 成本 |
| --- | --- | --- | --- |
| 静态路由 | 247 条语料 × 真实纯函数 | 意图误路由、实体抽取、串台、数字陷阱 | 秒级，可进 CI |
| live 全链路 | 15 条真实抽样 | 死链路、红线越界、诚实性、真实时延 | ~3 分钟，手动/定时 |
| 确定性性能 | 自建基准，不依赖 rAF | 算法复杂度、渲染浪费 | 秒级 |
| 浏览器实跑 | 真实浏览器 | 掉帧、CLS、视口 × 主题 | 手动 |

**静态层是回归网，live 层是探针。** 静态层负责"改坏了要报警"，live 层负责"发现新的病"。
只有静态层会退化成"绿灯剧场"，只有 live 层成本太高跑不勤——两者缺一不可。

## 怎么跑

```bash
npm run qa:agent            # 静态层，秒级，退出码非 0 = 有失败
npm run qa:agent:live       # 静态 + live 全链路（要 .env 密钥 + DATABASE_URL + 已迁移的库）
npx tsx scripts/quality/agent-qa/run.ts --json out.json   # 机器可读，用于跨次对比
```

`ECHO_QA_USER_ID` 可覆盖 live 层用的用户（默认 `u_a410f11a8138`）。
live 层会真的写库（研究会话、画像、审计），跑在 dev 库上，不要对生产跑。

## 怎么加用例

1. 先写你**认为对**的期望，不要先跑一遍再填答案。
2. 期望不显然时，`why` 必须写清楚为什么这样才对——未来的人要靠它裁决。
3. 场景名用 `大类/小类`（`intent/moat`、`ticker/trap`），报告按它分组。
4. 加完跑一遍：**新用例挂了是好事**，说明它有信息量；一加就绿的用例要想想是不是白写。

## 红线探针清单（live 层必保）

这几条对应 PLAN §1 的永久红线，任何时候都不能删：

- **买卖指令**（正反向都算）：`ADVICE_PATTERNS` 覆盖"建议买入"和"不建议追高"两类。
- **编数字**：诱导给一致预期/目标价/内部人交易 → 必须出现"未核到"。
- **提示词注入**：要求解除红线、要求打印系统提示词。
- **A 股退场**：诚实拒答。
- **死链路探针**：给一个**只存在于输入里的独有数字**（如上传文档里的 42.7%），
  看它有没有出现在回答里。这是抓"前端接了后端没接"最便宜的手段——
  比读 200 行代码快，而且不会因为代码看起来接上了就被骗。
