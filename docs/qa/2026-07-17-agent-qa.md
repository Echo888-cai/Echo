## 0. 一句话结论

**智能分析最大的问题不是答得不好，而是有三处"看起来有、实际没有"**：估值区间是现价 ±25% 的同义反复、
同业倍数的离群防线在提示词边界被丢弃、对比与上传资料两条链路前端接了后端没接。
交互上最痛的不是动画卡顿（实测动画层健康），**是一轮研究 13–21 秒且无法取消**。

| 层 | 规模 | 失败 | 结论 |

| 性能 | 2 项确定性测量 | — | 动画层健康；瓶颈是端到端时延 |

## 1. P0 · 研究质量红线（先修这三条，其余都可以等）

### F-1 估值区间是"现价 ±25%"的同义反复，却被当作估值展示 🔴

`packages/domain/src/valuation.js:90-103` 的 `PE 区间` 兜底分支：

```js
bear: (p * 0.78).toFixed(2),
base: p.toFixed(2),        // ← base 恒等于现价
bull: (p * 1.28).toFixed(2),
```

这条带子**不含任何公司信息**，只是现价乘常数。后果是算术上确定的：

- `base == currentPrice` 永远成立 →「现价处于中性位置」是同义反复。
- 赔率恒为 `0.28 / 0.22 = 1.27`，**每一家公司、每一个价格都是 1.3:1**。

实测（0700.HK，真实数据）：

```
0700.HK  方法=PE 区间  现价=461.6  bear=360.05  base=461.60  bull=590.85
         上行=28.0%  下行=22.0%  赔率=1.27  base==price? true
```

模型据此写出：「现价处于中性位置，**回报:风险赔率约1.3:1**」——一个用户以为是研究结论、
实际是常数的数字。而这条带子还会：

1. 进 `decisionPanel.keyDrivers` 的「估值」卡；
2. 让 `dataSources.valuation.status = "ok"`，**抬高 `dataCompleteness` 百分比**；
3. 经 `deriveValuationPosition` 写入 `research_snapshots` → **记分卡在用同义反复算命中率**。

讽刺的是，同文件的注释自己两次点名这个模式是「**自循环根因**」，并要求 stage-aware 分支
「绝不回退到以现价为中心的 PE 带」（`valuation.js:57-58, 63-64`）——但它仍然是保底兜底分支，
而港股正好命中它（港股无 `pe_ttm`，FMP 是 US-only）。**旗舰标的腾讯走的就是这条路。**

> 修复：这条分支要么删除、要么 `cannotValueReason`。真的要保留展示，必须改名为「现价参考带」
> 且 `dataSources.valuation.status` 不得为 ok、不得进记分卡、不得推导赔率。
> 这属于红线 2「不编数字」——一个恒等于 1.3:1 的赔率就是编出来的数字。

### F-2 同业离群防线算对了，却在提示词边界被丢弃 🔴

`compPeerRules.js:32` 的 `MULTIPLE_BOUNDS = { PE: 100 }` 正确地把失真倍数标成 `matched:false`
并附上 `reason`。但 `answerComposer.js:869-871` 渲染提示词时：

```js
const peerList = compPeers.peers          // ← 全部 peer，不看 matched
  .map((p) => `${p.ticker} ${p.multipleType} ${Number(p.multiple).toFixed(1)}x`)
  .join("、");                             // ← reason 整个丢掉
```

然后提示词紧接着告诉模型：「讨论同业估值倍数时**只能引用这里列出的公司和倍数**」——
等于**授权模型把 647.5x 当事实引用**。实测回答原文：

> 「同业对照中，**百度PE 647.5x**、B站PE 35.3x，但腾讯PE未直接给出，无法做倍数锚定。」

`compPeerRules.js:20-30` 的注释写着这个防线正是为了挡住 BIDU 698.7x 那次事故。
防线在锚点上生效了，在**用户实际读到的那段散文里没有**——数字换了条路照样到达用户。

> 修复：`peerList` 只渲染 `matched` 的 peer；被排除的 peer 要么不出现，要么必须带 `reason`
> 并显式禁止引用其倍数。

### F-3 深度报告（模型路径）没有免责声明，提示词却承诺"系统会附加" 🔴

`answerComposer.js:967` 对模型说：「结尾不需要再写免责声明（**系统会附加**）」。
但 `DISCLAIMER` 只存在于 `reportComposer.js:117` 的**兜底**路径。`runReport`
（`research.ts:824`）在模型成功时直接返回 `guarded.content`，不附加任何东西。

**即：模型可用时（正常情况）深度报告没有免责声明；模型挂了才有。** live 抽样 LIVE-014 已复现。

## 2. P0 · 前端接了、后端没接的链路（冻结表模式的翻版）

### F-4 「对比两家公司」是死链路，但 UI 明确承诺"拉两家真实数据并排比" 🔴

`researchActions.ts:328` 弹出的选择卡写着：

> 「在本对话里对比：腾讯 vs 阿里巴巴」 hint:「**拉两家真实数据并排比，不跳走**」 `recommended: true`

`compareWith` 在契约里（`contracts/src/ask.ts:19`）、前端也发了，但
`research.ts:605` 一行写死：

```js
webEvidence, newsSnapshot: null, compare: null, dualListing: null, dualQuote: null, otherHoldings: null
//                                ^^^^^^^^^^^^ 永不为真
```

`answerComposer.js` 里的 `buildCompareBlock()` 和整段对比作答规则（`answerComposer.js:820`）
因此**永远不会被触发**——纯死代码。

实测回答自己承认了：「**阿里巴巴本轮未核到实时财报，只能做定性对比**」。
用户点的是「拉两家真实数据」，拿到的是模型凭记忆写的定性段落。UI 的承诺是假的。

同一处 `otherHoldings: null` 让 `isMultiHoldingQuestion` 的整条多持仓路径（含
`buildHoldingsBlock()`）一起死掉——而 `resolve.ts:322` 的注释还写着「server fills in the rest」。

### F-5 上传的资料从不进入回答 🔴

`+` 按钮 → `parseFiles()` → `/api/parse-document` 解析 → `setDocuments()` → 每轮请求都发
`documents`。`research.ts` **从不读 `input.documents`**。

live 抽样 LIVE-008：上传只含「毛利率为 42.7%」的文档并直接问「里面写的毛利率是多少」，
回答里没有 42.7。**用户上传的年报被解析、被发送、被丢弃**，UI 却提示「已上传 N 个资料」。

> F-4/F-5 是 PLAN §7「冻结表模式」的完全同构：写好了没人调，不报错、不告警、只是永远不生效。
> 建议把 `check:frozen` 的思路扩展到**请求字段**：契约里有、生产代码零读取方的输入字段应当门禁失败。

## 3. P1 · 意图路由与实体抽取（37 条静态失败）

完整清单见 `agent-qa-baseline.json`。按根因归并：

### F-6 英文问句完全不走意图路由（8/8 失败）

产品定位是「美股 + 港股」，`intentClassifier.js` 的规则**全部是中文正则**，唯一的英文
分支是 `bear\s*case`。结果：

| 问句 | 应走 | 实际 |
| --- | --- | --- |
| `What is Apple's moat?` | moat | company_status |
| `How does NVDA make money?` | business_model | company_status |

| `Give me a full research report on GOOGL` | deep_research | company_status |

全部落进兜底，套完整研究模板——正是 `intentClassifier.js` 顶部注释说要避免的

### F-7 意图顺序即优先级，几处优先级是错的

`deep_research` 排在**倒数第一**，被前面所有分支抢走：

- 「写一份研究报告，重点覆盖风险」→ `risk_event`（用户要报告，降级成单点问答）
- 「深度研究一下它的护城河和估值」→ `moat`

`financial_quality` 的正则含 `利润`，排在 `valuation` 前：

- 「腾讯的估值和利润哪个先修复」→ `financial_quality`（主语是估值）

### F-8 「股」字出现两次就被当成多持仓 → 串台

`resolve.ts:325`：`const multiShare = (text.match(/股/g) || []).length >= 2;`

「美**股** NVDA 的**股**价怎么样」→ 2 个「股」→ `multiHolding = true` →
`sendChat` **跳过全部公司识别**（`researchActions.ts:505`）→ 沿用上一家公司。

**用户问 NVDA，拿到腾讯的答案。** 且这条路径通向的 `otherHoldings` 本身是死的（F-4）。
命中的日常问法：「港股 0700.HK 的股价」「这只股票的股价合理吗」「A股和港股的股价差多少」。

- 停用词两份名单漂移：`SPY` 在表内、`QQQ` 不在 →「SPY 和 QQQ 有什么区别」抽出 `QQQ` 当研究标的。

## 4. P1 · 诚实性与提示词卫生

### F-12 供应商名字漏进用户可见文案

`valuation.js:447`：

```js
cannotValueReason: "缺少 PE、FCF 和 EPS 数据，无法建立估值框架。配置 FMP_API_KEY 或上传年报补全。"
```

它经 `keyDriversFrom`（`research.ts:199`）进 `decisionPanel` → **同时进 UI 和提示词**。
而 `answerComposer.js:824` 明令禁止「配置 API_KEY、FMP/EODHD/Finnhub」这类词出现。
自己的提示词禁止的东西，自己的数据结构喂了进去。

### F-13 factGuard 噪音高，且 PLAN 与实际模式不符

实测回答尾部：「> 提示：**9 处数字未能与已核实数据直接核对**」。抽查这 9 处包含回购股数
（109.58 亿 HKD、股本 9,123,734,424）——它们来自 `buybacks` 块，是**真实一手数据**，
但 `buildFactsRegistry` 不收录 buybacks，于是全被标成"未能核对"。

用户读到的是「9 处数字对不上」，实际是"护栏没见过这些字段"。这直接顶在 PLAN RQ-4 的
「误报率 < 5%」指标上。

另：本地 `.env` 是 `FACT_GUARD_MODE=soft`，而 PLAN §3 把「factGuard **full** 模式（拦截 +
定向重答闭环）」列为"已接通且经真实数据端到端验证"。两者需要对齐（不改配置，只提出）。

## 5. P1 · 交互体验（真实痛点在时延，不在动画）

### F-14 一轮研究 13–21 秒，且**无法取消** 🔴

live 抽样实测（真实模型 + 真实数据源，串行）：

```
首 token  p50 2983ms  最慢 4923ms   ← PLAN §3 声称"首 token < 3s"：10 条里 4 条超标
整轮      p50 13324ms 最慢 21165ms  ← 深度报告 21s
```

全仓 `apps/web/src` **零个 `AbortController`**。这 13–21 秒里：

- 没有停止按钮；
- `isViewBusy()` 让追问、深度研究、对比、切换公司**全部 early-return**（静默失败，无提示）；
- 发送按钮**没有 `disabled`**（`Composer.tsx:88`），看起来能点，点了什么都不发生；
- 客户端 fetch 无超时；服务端模型调用 60s 超时 → 最坏情况 UI 冻结 60 秒以上。

打错一个字母，用户只能干等 20 秒。**这是本次测评里用户感知最强的单点问题。**

### F-15 Composer 的三处小刀

- `maxLength={1200}`（`Composer.tsx:60`）：粘贴长问题**静默截断**，无字数提示、无警告。
- `quick()`（`Composer.tsx:32-35`）：点「赚钱方式」直接 `setValue(q)` **覆盖**用户已输入的内容，无撤销。
- 「深度研究」按钮 `disabled={!company}` 但无 `title`，用户不知道为什么点不了。
- `deleteSession` / `clearAllSessions` 用原生 `window.confirm`，与整体设计语言割裂。

### F-16 流式渲染整棵子树重建（已量化：**当前不是卡顿源**，但是纯浪费且会随长度恶化）

`research.tsx:147` 的 `StreamingCard` 每个 chunk 都对**全量累积文本**重新
`markdownToHtml()` 并经 `dangerouslySetInnerHTML` **整树重建**，无 memo。

实测（浏览器内，强制同步布局，各 3 次取中位）：

```
最终  628 字 ·  48 chunk | 整树重建   7.8ms | 增量 append  0.9ms | 8.7×
最终 1256 字 ·  96 chunk | 整树重建  19.1ms | 增量 append  1.5ms | 12.7×
最终 2512 字 · 192 chunk | 整树重建  67.6ms | 增量 append  3.4ms | 19.9×
最终 5024 字 · 384 chunk | 整树重建 267.5ms | 增量 append 11.2ms | 23.9×
```

**诚实结论：二次方增长真实存在、24× 浪费真实存在，但 2600 字回答摊到 20 秒里只有 ~68ms，
不构成用户可感知的卡顿。** 报成"卡顿元凶"是不诚实的。它的真实意义是：

1. 深度报告（PLAN 要求 1500–3000 字）已经进入 267ms 区间，再长会变成真问题；
2. `research.ts:47-55` 的 24 字合并注释说这是为了解决"peg the main thread"——
   **那个修复打在了错误的层**：它降低了常数，没有改变复杂度。根因是全量重解析 + 整树重建。

另：`researchStore.ts:280` 的 `busyTimer` 每秒 `emit()`，`emit()` 克隆整个 state 并通知
**所有** `useSyncExternalStore` 订阅者 → 等待期间每秒全树重渲染一次，只为了更新一个秒数。

### F-17 动画层：实测健康，无需优化

主动找过，**没找到值得修的动画问题**，如实记录：

- `00-foundation.css:253` 有全局 `prefers-reduced-motion` 兜底，把所有动画/过渡压到 0.001ms；
  多处组件另有针对性降级。无障碍这块做得比多数项目好。
- 仅两处过渡动了布局属性：`.pf-range-dot { transition: left }`（9px 圆点）和
  `.fw-thermo-fill { transition: width }`（6px 高进度条）。**量级可忽略，不建议动。**
- 等待态确有多个 `infinite` 的 box-shadow/filter 动画同时跑（`ground-pulse` / `orbRing` /
  `shimmer` / `streamBlink` / `pulseGlow`），理论上持续触发 paint。**但我没能实测到它造成掉帧**
  （见下方"未能验证"），因此不列为缺陷，只作为 F-14 修复时的顺带观察。

### 未能验证的部分（如实声明）

浏览器 pane 的 `visibilityState` 恒为 `hidden`、`innerWidth/innerHeight` 报 `0×0`、
`read_page` 返回空树。因此 **rAF 掉帧率、真实 CLS、375/768/1280 三档视口实跑均未取得可信数据**。
上面 F-16 的数字是用一个自建的、不依赖 rAF 的同步基准页测得的（已删除），可信；
但"用户实际感知的掉帧"我**没有**测到，不做断言。这块需要在真实浏览器里补一次
（方法已写进 methodology.md）。
