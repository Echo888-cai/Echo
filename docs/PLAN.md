## 0. 一句话论点（接手先读这条）

v1–v4 把判断生产线建到可信、把日常面（持仓/看盘/移动端）立了起来，CN 轨把覆盖面补到港美 A 三市——产品第一次**值得开给别人用**。但"开给别人用"的三堵墙全在平台层，且互相独立于业务功能：**数据模型是单用户的**（`portfolio_positions` 主键就是 ticker，两个人买同一只票会互相覆盖成本价）；**零鉴权且静态文件兜底会把 `.env` 和整个 `luvio.db` 发给任何请求者**（今天只绑 127.0.0.1 所以没爆，公网暴露第一天就是事故）；**成本与配额无护栏**（一个好奇用户一下午能烧光当月 Tavily 免费额度）。**U 轨 = 用最小改造把单机研究工具变成 10 人邀请制 beta**：安全底座 → 鉴权 → 多租户迁移 → 部署 → 配额，全程保持 Node + SQLite + 零新增运行时依赖（红线 1）。SQLite 不是瓶颈（WAL 撑 50–100 用户没有问题，换 Postgres 的触发条件记录在 §2 E14 备注，不许提前）。业务迭代候选池（§1.2 的 R/W 系列）记录在案，**由 beta 真实反馈定优先级**（红线 7），本轨只做 beta 第一印象必需的 P14。

## 1. 诊断依据（2026-07-10 技术 + 业务双视角审计）

### 1.1 技术审计四发现（开放 ≤10 人的真实阻断项）

**发现① · 数据模型是单用户的（最硬的墙）**。逐表核对 `src/db/migrations/001_init.sql`：`portfolio_positions` 主键 = ticker（`:123`，无 user 维度，第二个用户录同一只票直接覆盖第一个人的成本/止损）；`watchlist_prefs` 主键 = ticker（`:208`，全局一张关注列表）；`research_sessions`（`:70`）、`company_profiles`（`:91`）无 user_id——A 的研究判断会渲染进 B 的界面。好消息：`watch_rules` / `notifications` / `profile_events` 三张表已有 `user_id TEXT NOT NULL DEFAULT 'local'` 脚手架（`:217` / `:137` / `:118`），当初留过口子，迁移模式可以直接沿用。

**发现② · 静态文件兜底会泄密（部署阻断项）**。`server.js:132-146` 的兜底逻辑对项目根目录下**任何可读文件**放行：请求 `/.env` → 全部 API key 原样发出；请求 `/luvio.db` → 整库（含未来所有用户的持仓与研究）可下载。路径穿越已被 `normalize`+`join` 防住，但根目录本身就有不该发的东西。今天 `server.listen(port, "127.0.0.1")` 兜着底，**公网暴露必须以静态白名单为前置**（新红线 19）。

**发现③ · 成本与配额无护栏**。每次 `/api/ask` 触发 LLM 调用 + 多路数据 API；≤10 用户共享同一组 key 意味着免费档限速被放大（Tavily 1000 次/月、FMP/Finnhub 免费档），scheduler 任务按用户迭代后再乘一遍。地基已有：`llm_audit`（migration 005）在记每次调用，缺的是每用户每日限额 + 聚合视图。

**发现④ · 部署面空白，但 SQLite 不是问题**。无 TLS（需反代）、备份只落本机（服务器挂 = 全没）、`/api/parse-document` 无请求体上限、无限速、cookie 登录态引入后需防 CSRF。**明确不是问题的**：better-sqlite3 同步写在本项目的短事务模式下 10 用户无感知；换 Postgres 属于过早优化，触发条件 = 多机部署 / 实测写争用 / 用户过百，三者一个没到。

### 1.2 业务审计（金融专家视角）：研究/看盘缺陷 → beta 候选池

> 定位判断先行：**"以判断状态盯盘 + 证伪条件自动核对"是真差异化**（行情软件和通用 AI 都做不了），M-3 治理 thesis 后这个灵魂才立住。以下按对价值投资者的实际伤害排序，**本轨不做**（P14 除外），beta 反馈定优先级。

- **R-a · 估值引擎行业盲区**（A 股接入后变紧迫）：PE/FCF/DCF 对银行/保险（该看 P/B、拨备、内含价值）、地产/REIT（NAV/FFO）、强周期（周期顶低 PE 是陷阱，该看正常化盈利/P/B 分位）会系统性误导；A 股种子池里就有银行和白酒。方向：valuationEngine 加行业路由，各自换锚，取不到就诚实说（红线 5）。
- **R-b · 盈余质量红旗**（A 股杀手级）：现有 profit-quality 只看"赚多少"（利润率/ROE），没看"赚得真不真"——应收/存货增速背离收入、经营现金流长期背离净利润、商誉占净资产、大存大贷。CNINFO 管道已把三表关键行拿到一手，**在一手数据上做红旗检查是边际成本最低、差异化最高的一步**。
- **R-c · A/H 溢价**（白捡的信号）：A+H 双重上市的溢价率是现成数据 + 现成双上市路由，估值锚直接有用，近似顺手项。
- **R-d · "还需核实什么"没有闭环**：数据完整度条诚实说了缺什么，但未决问题随对话滚走。画像侧加"未决问题清单"：研究时沉淀 → 下轮自动带出 → 数据到货提醒。
- **R-e · 行业中间层缺失**：有公司层和宏观层，没有行业层（半导体周期/新能源价格战/白酒库存）；持仓相关性往往发生在行业层。可从组合体检卡按主题聚合暴露起步。
- **W-a · 判断状态机缺温度计**：逻辑还在/有风险/已证伪目前靠触线和财报到货的离散事件翻转；基本面恶化是渐进的（毛利率连续三期向证伪线漂移时系统沉默）。方向：证伪线加"距离 + 趋势"预警。
- **W-b · 业绩期模式**：缺季报季一屏视图——"本周关注股 4 家出财报，各家共识预期 / 我的证伪线 / 上次 beat/miss"。素材（earningsCalendar + falsifyRules + researchReview）全部现成，纯组装。
- **W-c · 关注列表是平的**：持仓/高关注/观察/退役混排；分组 + 按"状态变化时间"排序（判断驱动的排序，不是涨跌幅排序）。
- **W-d · A 股特有信号缺位**：限售解禁日历、股权质押、商誉减值预警、北向变动——A 股语境的证伪级事件；CNINFO 公告流已通，解禁/质押解析是增量不是新管道。
- **W-e · 通知信噪比**：现在按单人偏好调校；多用户后需每用户通知偏好（等级/通道），否则"安静的研究札记"变吵闹推送源，与品牌承诺直接冲突。→ 并入 P14。

**北极星指标（beta 前定好，防止 10 个人把方向拽向 10 边）**：建议 = "每周主动打开 ≥3 天的用户比例"，辅助 = 盘前速报点开率。

### 1.3 合规边界

腾讯/新浪行情不可商用（红线 6）→ **beta 必须免费、邀请制、不公开宣传**（新红线 17），直到拿到可商用行情授权。10 人朋友内测在此边界内。

## 2. Gap 清单（编号承接：工程 E11 起，产品 P14 起，研究员 R13 起）

**E11 · 安全底座** ★（发现②④前半）
- 静态服务改白名单（只发 `index.html` / `src/ui` / `src/styles` / 图标；`.env`、`*.db*`、`docs/`、`src/server/` 永不可达）；请求体上限（parse-document 单独放宽到合理值）；每 IP/每用户内存令牌桶限速（零依赖）。
- **优先级**：P1（一切部署的前置，红线 19）｜ **工作量**：≈1 天。

**E12 · 鉴权与邀请制** ★（发现②后半）
- `users` 表 + 一次性邀请码 + `node:crypto` scrypt 口令哈希 + 签名 HttpOnly cookie（`Secure` + `SameSite=Lax` + 自定义头 CSRF 校验）。零新增依赖（红线 1）。所有 `/api/*` 过鉴权中间层；未登录只见登录页。
- **优先级**：P1 ｜ **工作量**：≈2-3 天。

**E13 · 多租户数据迁移** ★（发现①）
- **公共数据层（无 user_id，全员共享省配额）**：companies / company_details / market_snapshots / hk_financials / cn_financials / web_evidence / earnings_* / comp_peers / insider_activity / hk_buybacks / historical_valuation / scheduler_state / llm_audit / fact_guard_audit。
- **私有层（加 user_id，用户拍板全私有）**：research_sessions（含 FTS 查询必须带 user 过滤）/ company_profiles / profile_events / research_snapshots / portfolio_positions / portfolio_snapshots / watchlist_prefs / watch_rules / notifications / documents。
- `portfolio_positions` 与 `watchlist_prefs` 主键改 `(user_id, ticker)`——SQLite 不能改主键，走"建新表 → 拷贝（user_id='local'）→ 改名"的非破坏性迁移（红线 9）；存量数据归属 Arlan 账号。scheduler 按用户迭代私有任务（触线/证伪/组合快照/速报），公共任务（备份/canary/业绩复核）全局一次。
- **优先级**：P1 ｜ **工作量**：≈2-3 天。

**E14 · 部署**
- 小型 VPS（2C4G 够用，**选香港/新加坡节点**——巨潮与腾讯行情海外机房访问不稳）+ Caddy 反代 TLS + `server.js` 继续只绑 127.0.0.1（Caddy 对外）。dbBackup 完整性校验通过后异地推送一份（rclone → B2/S3，运维工具不算运行时依赖）。生产 doctor 检查项补齐。
- **换 Postgres 的触发条件（记录在案防重提）**：多机部署 / 实测写争用 / 用户过百；到达前不动。
- **优先级**：P1 ｜ **工作量**：≈1-2 天。

**E15 · 每用户配额与成本**（发现③）
- llm_audit 按 user 聚合 + 每用户每日调用/成本限额（超限诚实提示"今日研究额度用完"，不静默失败）。E10 成本视图已在 M-4（PR #22）落地，本阶段只加分用户维度与限额。
- **优先级**：P1 ｜ **工作量**：≈1-2 天。

**P14 · Beta 第一印象**（兄弟主刀，与 U-1/U-2 并行）
- Onboarding（首次登录的 3 步引导：问一家公司 → 加关注 → 记一笔持仓）；空状态全检（新用户每页第一眼不能是空白或报错）；应用内反馈入口（一行文本 + 自动带上下文落库）；每用户通知偏好（W-e）。
- **优先级**：P1 ｜ **工作量**：≈3-4 天。

**候选池（beta 反馈定优先级，编号从此起不重排）**：P15 业绩期视图（W-b）、P16 关注列表分层（W-c）、R13 盈余质量红旗（R-b）、R14 估值行业路由（R-a）、R15 A/H 溢价（R-c）、R16 未决问题闭环（R-d）、其余 R-e / W-a / W-d 记录在案。**PWA（原 P12）按既定决策放 U 轨之后**——beta 用户装进手机桌面前，先保证登录态和私有数据边界存在。

## 3. 下一阶段路线：U 轨（User——多用户 Beta）

| 阶段 | 内容 | 合并的 gap | 谁 | 预估 |
|------|------|-----------|----|------|
| **U-0** | **家务 + 安全底座**：合并 PR #22（M-4，见 §5 注记）→ `git pull` 对齐 → CN 轨合入 main（branch+PR）→ 静态白名单/限速/请求上限 | E11 | Arlan | ≈1-2 天 |

全轨 ≈2-2.5 周（两人并行）。**顺序理由**：U-0 是一切的前置（红线 19），且 CN 轨压在工作区一天就多一天合并冲突风险；U-1→U-2 有依赖（迁移需要 user 概念存在）；U-3 放在 U-2 后是为了第一次公网暴露时私有边界已存在；P14 全程无后端依赖可并行；U-5 前两人各自用真实账号在生产环境完整走一遍研究→关注→持仓→通知链路（真实数据验证惯例）。

### U-2 关键规格（多租户是本轨风险最高的一步）

- 迁移全部非破坏性（红线 9）：每张私有表 `ALTER TABLE ADD COLUMN user_id TEXT NOT NULL DEFAULT 'local'` + 建索引；两张主键表走重建三步（见 E13）；迁移前 dbBackup 强制跑一次。
- **repository 层每个查询显式带 user_id 参数**——不做隐式全局 current-user 单例（测试和 scheduler 都要能以任意用户身份调用）。code review 硬检查：私有表查询缺 user 过滤 = 阻断合并（红线 18）。
- `tests/phase-u2.mjs` 必须包含**跨用户隔离用例**：用户 A 写入持仓/关注/会话后，以用户 B 身份逐接口断言不可见。

## 3.5 双人协作标准（用户定调分层分工，全轨适用）

### 职责划分

| | **Arlan — 平台与数据 Owner** | **兄弟 — 产品与体验 Owner** |
|---|---|---|
| **独占领域** | U 轨全部（鉴权/迁移/部署/配额/安全）、数据管道（CNINFO/HKEX/EDGAR）、scheduler、DB 迁移、factGuard 升档观察 | P14 及后续 UI 迭代（P15/P16 候选）、分享图（P13）、PWA（U 轨后）、§3.5 视觉验收执行、beta 反馈收集与每周归并 issue |
| **代码属地** | `src/server/**` `src/db/**` `scripts/` `src/*.js`（数据适配层） | `src/ui/**` `src/styles/**` `index.html` |
| **共同持有** | `docs/`、业务逻辑服务（valuationEngine/eventEngine 等——谁的阶段需要谁改，对方 review） | 同左 |
| **值守** | 生产报警（Telegram 错误通道），beta 开放后按周轮换 | 同左轮换 + beta 用户"人肉客服" |

### GitHub 机制（细则同步写入 docs/GITHUB_WORKFLOW.md）

1. **保护 main**：禁止直接 push（纯文档除外，沿用现规则）；PR 必须 CI 绿（lint + typecheck + 全量测试设为 required checks）。
2. **互审**：对方 approve 才可合并；沿用已确立的低风险自动合并政策，但 **DB 迁移 / 鉴权 / 部署配置 / PLAN.md 改动必须双人 approve**——PLAN.md 是宪法，改宪法要全体同意。
3. **CODEOWNERS**：按上表代码属地落 `.github/CODEOWNERS`（待兄弟 GitHub 账号确定后落地），开 PR 自动请求正确 reviewer。
4. **分支命名**：`feat/u1-auth`、`fix/watch-xxx`、`docs/xxx`——轨道编号进分支名；commit 沿用中文单行惯例。
5. **一块 GitHub Projects 看板**：待办/本周/进行中/待 review/完成；每个 PLAN 阶段一个 milestone。**PLAN.md 仍是唯一权威，看板只是镜像**。
6. **节奏**：周一 30 分钟定各自本周阶段目标；周五互相 demo + 完成项写回 §5 状态表。PR 小步（软上限 ≈400 行 diff），大阶段拆成可独立合并的 PR。

## 4. 明确不做 / 红线（承接 v1–v4 全部条目，新增 17–20）

1. **不换前端框架**：vanilla ESM + 事件委托。
2. **不做完全自主 ReAct 循环**：规划步数 ≤3，规则优先。
3. ~~不做 A 股~~ **已反转（2026-07-10）**：以港股同等工程质量纳入 A 股（主板+创业板核心公司，`src/data/cnStocks.js`），巨潮一手管道（`cnFilingsPipeline.js`）+ 腾讯/新浪双源行情已实测跑通，66 支种子 91% 有一手财报。
4. **不给买卖指令**：合规红线，宪法约束。
5. **不编数字**：取不到就"未核到"，禁止估计范围/反推。
6. **不在没有授权前把腾讯行情当商用数据源**（商业化阻断项；A 股新浪源同此）。
7. **不做过早抽象**：候选池等真实 beta 反馈；EA-6 等真实用户。
8. **canary 不进 CI**：CI 保持无 key、不烧配额。
9. **涉及真实用户数据的 schema change 必须走非破坏性 migration**（U-2 是最大适用对象）。
10. **factGuard 升档必须以落库的真实误报率为依据**。

## 6. 怎么跑 + "完成"的定义（速查，承接 v1–v4）

```bash
npm install                 # 只有 better-sqlite3 一个原生依赖
npm run seed                # 建/重置本地 SQLite 种子库
npm run dev                 # http://127.0.0.1:4173
npm test                    # 全量用例必须全绿（EXIT=0）再提交
npm run lint && npm run typecheck   # 0 error（已进 CI，勿回潮）
npm run doctor              # 能力体检；--live 才发网络探活
npm run canary              # 真实数据 canary（本机/scheduler 职责，不进 CI）
# 后端无热重载！改 src/server/** 或 src/*.js 后要重启 node
# 隔离测试：LUVIO_DB_PATH=$TMPDIR/x.db PORT=4199 node server.js
```

**一次改动的"完成"= 代码 + 对应测试（进 `tests/`，接入 `npm test`）+ §5 状态表标 ✅ + 一条中文 commit + 对方 review（§3.5）。** 浏览器可见的改动必须实跑验证（含红线 16 三档视口 × 双主题）；涉及外部数据源的改动必须真实调用验证；**U 轨新增：涉及私有数据的改动必须包含跨用户隔离用例（红线 18）**。代码改动走 branch + PR 进 main；纯文档改动可直接 commit main（PLAN.md 除外——双人 approve，§3.5）。

**key 都在 `.env`（gitignored）**：GLM / DEEPSEEK / FMP / FINNHUB / TWELVEDATA / TAVILY。沙箱会墙搜索引擎，端到端 web 证据效果要在本机/生产验证。
