## 0. 一句话论点（接手先读这条）

v1–v3 把**判断生产线**建到了可信：数字有锚（同业/日历/分位/一手回购与内部人）、有护栏（factGuard）、有留痕（画像/快照/审计表）、有兑现（业绩闭环 + 基本面证伪自动核对）。但这些能力只在"研究时"发光——**用户每天打开产品的场景是持仓和看盘，而这个日常面实测有硬伤**：移动端看盘列表是坏的（375px 下公司名不可见、财报徽章压在现价上）；看盘的差异化灵魂"以判断状态而非价格盯盘"被上游架空（6 只关注股里 5 只的"投资主线"存的是数据碎片）；持仓是一张会被对话推走的聊天卡片，组合是全产品唯一没有时间维度留痕的资产。**M 轨 = 把研究能力兑现成每天想打开的产品**：持仓一级页面打头，移动端修复、看盘内容质量、PWA 收尾。增长与商业化在多用户化之前结构性无解，本轮明确不动（§1.1、§4）。

## 1. 审计方法与证据基础

### 1.1 五角度战略评估（为什么下一程是 M 轨）

- **投资研究质量**——已是长板，进入维护模式。继续加码边际收益递减；factGuard shadow→soft 升档是"等真实样本"不是"建功能"（红线 10），列为持续观察项而非阶段。唯一值得主动做的是画像 thesis 治理（R12）——名义是研究资产质量问题，收益却兑现在看盘页上。
- **移动端体验 + 持仓/看盘**——**合并为同一件事，当下杠杆最高**。看盘/持仓天然是移动场景（盘前扫一眼、收到证伪/业绩通知点进来看），通知通道已建成（通知中心 + Telegram），缺的恰是"点开通知后落地的那个页面在手机上能看"。实测这个落地面是坏的（§1.2 发现①）。→ 本轨主战场。
- **数据护城河**——方向已对：真正的护城河是**自沉淀的时间序列**（`market_snapshots` 逐次研究的真实 PE、`fact_guard_audit` 误报率、`hk_buybacks`/`insider_activity` 一手序列、画像/快照/记分卡），跑得越久越值钱且买不到现成的。本轨加一个便宜加速器：**每日组合快照**（E9，随 M-1），同时是净值曲线的数据源——时间序列等不来，越早开始积累越好。商用数据底座只能是 HKEX/SEC 一手（红线 6 腾讯行情不可商用），这条路已在走。
- **用户增长**——结构性未解锁：单机、单用户、无 auth、无部署。顺序不能倒：**留存面（M 轨）→ U 轨（部署 + auth + 多用户）→ 增长**。低成本铺路动作只留一个：分享图导出（P13 顺手项，零后端的口碑素材）。
- **商业化**——五者最晚，维持 v2 以来 P8 缓做判断。三个前置（行情授权/多用户底座/单位经济模型）一个没到。现在唯一该做的动作：研究成本汇总视图（E10 顺手项）——llm_audit 已在记每次调用，把每轮研究的真实成本算清，零新增留痕，等定价那天直接有数。

### 1.2 实跑 UI 审计（2026-07-07，桌面 + 375px 移动端浏览器实测）

**发现① · 移动端看盘列表布局损坏（bug 级）**。375px 实测：公司名整列不可见；"财报 2026-08-11"徽章直接压在现价"452"上；顶部导航"研究/看盘/设置"换行成竖排两行。根因已定位：`.wl-main` 虽有 `min-width:0`（`06-watch.css:200`），但 `.wl-main > * { flex: none }`（`:202`）让 ticker/市场徽章/财报徽章全部不可收缩，只有 `.wd-name` 可收缩（`:203`）——窄屏 4 列网格（`:263`）下名称被挤到 0 宽（所以公司名"消失"），不可收缩的徽章溢出 flex 容器（无 overflow 裁剪）压进相邻数字列。
**发现② · 看盘的差异化灵魂被上游数据质量架空**。看盘以"判断状态"（逻辑还在/有风险/已证伪）为第一信号，这是对行情软件的真正差异化——但实测 6 只关注股里 **5 只的"投资主线"存的是数据碎片**（如"收入增速 -1.10%，毛利率 55.71%"），只有 1 只是真正的主线陈述。`WL_BAD_THESIS`（`src/ui/watch.js:67`）能挡失败占位文案，挡不住"看起来像数据"的碎片——问题在画像沉淀 thesis 的上游，不在前端。另：公司页"近期事件"同一条腾讯回购新闻出现 3 个变体（46.50万股/47万股/46.5万股）不去重。
**发现③ · 持仓的信息架构与看盘不一致**。`showPortfolio`（`src/ui/portfolio.js:6-17`）把持仓渲染成研究对话流里的一条 assistant 消息：看盘页点"我的持仓"→ 跳到研究页 → 对话里塞一张卡 → 聊几轮就被推走。组合体检卡（纪律红线/集中度/跨币种折算）内容本身很好，**载体不对**——"我的钱现在怎么样"是每天都要问的问题，值得一个稳定的一级入口。
**发现④ · 组合没有时间维度**。只有此刻快照（现价/浮盈/距触线），没有净值曲线、没有历史。研究侧三层留痕（会话/快照/画像）齐备，**组合恰是全产品唯一没留痕的资产**——"我的判断 vs 我的组合表现"无从回看。

## 2. Gap 清单（七要素，编号承接：研究员 R12 起，产品 P9 起，工程 E9 起）

> 字段约定与优先级定义同 v2/v3（P1 = 下一程就做；P2 = 第二程候选；P3 = 记录在案等时机；P4 = 明确缓做）。

**P9 · 持仓是会滚走的聊天卡片，组合没有时间维度** ★（发现③④）
- **问题**：持仓无独立页面/路由，以对话消息为载体；组合无净值曲线、无每日留痕；已实现盈亏无从谈起（现有 `portfolio` 表只存当前持仓，没有交易流水模型）。
- **为什么重要**：持仓是日频最高的查看对象；组合时间序列是"研究是可复盘资产"叙事的最后一块（判断有快照，组合没有）；每日快照同时是数据护城河的自沉淀序列。
- **用户价值**：高（日常主入口）。｜ **工程风险**：低（纯本地，复用现有 `/api/portfolio` + `/api/portfolio/review`，新增一表一任务）。
- **明确不做**：已实现盈亏（需要交易流水数据模型，等真实需求，不为一条曲线重构记账模型）。
- **优先级**：P1（用户定调打头）｜ **真实数据验证**：不需要 ｜ **工作量**：≈3 天。

**P10 · 移动端看盘列表布局损坏** ★（发现①）
- **问题/根因**：见 §1.2 发现①，file:line 已定位。
- **为什么重要**：这不是"体验待优化"，是"打开就是坏的"；移动端是看盘的天然场景。
- **优先级**：P1 ｜ **真实数据验证**：不需要（375px 实跑截图验收）｜ **工作量**：≈1 天。

**R12 · 画像 thesis 沉淀的是数据碎片——看盘差异化被架空** ★（发现②前半）
- **问题**：thesis 字段存入"收入增速 -1.10%，毛利率 55.71%"类碎片（实测 6 只中 5 只），列表与公司页的"投资主线"因此没有信息量。
- **方法（套 F-3 成熟模式）**：**不做事后文本解析**——研究时让模型结构化输出一句主线（同 `FALSIFIERS_JSON` 的剥离机制），画像侧对 thesis 字段加碎片过滤器（指标名开头/数字密度阈值 → 拒收，保留旧值或置空）；存量脏 thesis 一次性置空待下轮研究自然重建，不硬迁。
- **优先级**：P1 ｜ **真实数据验证**：需要（真实模型输出的主线质量）｜ **工作量**：≈1.5 天。

**P11 · 同题材事件不去重**（发现②后半）
- **问题**：同一回购事实的多个新闻变体并列展示，稀释公司页信息密度。
- **方法**：eventEngine 侧同 ticker 同日标题归一（数字/单位归一后相似度）聚合为一条 + "同题材 n 条"折叠；**宁可漏合并不可错合并**（不同事件误合并会丢信息）。
- **优先级**：P1（与 R12 同阶段）｜ **真实数据验证**：需要（真实新闻流的去重准确率）｜ **工作量**：≈0.5 天。

**P12 · C2 兑现：PWA**（v2 以来缓做，本轨解锁）
- **为什么现在做**：M-1/M-2/M-3 之后移动端日常面才值得"装进手机桌面"；通知点开深链落地页已修好。
- **优先级**：P2（本轨收尾）｜ **真实数据验证**：不需要（真机安装实测）｜ **工作量**：≈1-2 天。

**E9 · 每日组合快照任务**（随 M-1，护城河加速器）
- **内容**：scheduler 第 8 个任务 `portfolio_snapshot`（每日 08:00，两市均已收盘，各用最近收盘价）→ 新表 `portfolio_snapshots`（migration 014：日期、分币种市值/成本/盈亏、持仓明细 JSON）。喂 M-1 净值曲线，也喂未来"判断 vs 组合表现"复盘。
- **优先级**：P1（含在 M-1 内）｜ **工作量**：≈0.5 天（含在 M-1 的 3 天内）。

**顺手项**：**P13 · 分享图导出**（研究卡/画像 → PNG，canvas 渲染，零后端传播素材，≈1 天）；**E10 · 研究成本汇总视图**（llm_audit 聚合进设置页，≈0.5 天）。
**持续观察（非阶段）**：factGuard shadow→soft 升档，等 14 天窗口的真实误报率达标（红线 10），不许拍脑袋。

## 3. 下一阶段路线：M 轨（Monitor/Mobile——日常面收口）

| 阶段 | 内容 | 合并的 gap | 预估 |
|------|------|-----------|------|
| **M-1** | **持仓一级页面**：路由 + 组合概览横幅 + 净值曲线 + 体检卡 + 持仓卡列表 + 每日快照任务 | P9 + E9 | ≈3 天 |

### M-1 规格（打头阶段，展开写）

- **路由与入口**：新增一级路由 `#/portfolio`。入口两处：看盘页"我的持仓"按钮改为**跳页**（不再往对话塞卡）；对话里的持仓意图**保留**（对话记账是产品特色，不砍），但渲染为紧凑摘要卡 + "打开持仓页 →"链接。**导航不加第四个 tab**——移动端导航宽度本就紧张（M-2 要修的就是它），用量起来后再议。
- **页面结构（自上而下）**：① **组合概览横幅**：分币种总市值 + 折算合计（口径显式标注"按 USD≈7.8 HKD 折算"）、当日盈亏、累计浮盈；② **净值曲线**：手写 SVG 面积图（复用公司页 pchart 模式），数据源 = E9 每日快照——起步阶段诚实显示"净值积累中 · 第 x 天"，缺日显示为断口，**不插值不回填**（红线 15）；③ **组合体检卡**：复用 `/api/portfolio/review` 现有数据（纪律红线/集中度/暴露条），样式升级为页面级卡片；④ **持仓卡列表**：升级现有 pf-card——点击进 `#/watch/:ticker` 个股页（打通持仓 → 看盘详情的回路）、距止损/止盈的可视化条、编辑/删除。
- **动效与视觉（§3.5 的首个适用对象）**：页面入场 fade + rise（沿用现有 calm motion 语言）；净值曲线一次性绘制入场动画（stroke-dashoffset，`prefers-reduced-motion` 降级为直接呈现）；盈亏数字 `tabular-nums` 防跳动；加载态用 skeleton 不白屏；375/768/1280 × 亮暗双主题实跑验收。

### M-2 规格

根因修复：徽章收缩策略 + `.wl-main` 溢出裁剪；窄屏（≤640px）行布局改**两行式网格**——第一行：状态点 + 公司名 + 徽章（名称恢复可见且省略号截断），第二行：迷你曲线 + 现价 + 今日涨跌（两行式腾出的宽度**恢复窄屏 sparkline**，v3 时代是直接藏掉的——好看优先）；顶部导航 375px 不换行（字距/内边距收窄）。验收：375px 实跑截图无重叠、无横向溢出、公司名可见。

### M-3 规格

thesis 治理：`prompts.js` chat 框架追加"一句投资主线"结构化输出行（同 F-3 的 `FALSIFIERS_JSON` 模式：抽取 → 校验 → **从可见正文剥离**）；`companyPortrait` 侧 thesis 字段加碎片过滤器（指标名开头 / 数字密度阈值 → 拒收）；存量 5 条脏 thesis 一次性置空待重建。事件去重：同 ticker 同日标题归一后聚合 + "同题材 n 条"折叠。真实数据验证：真实发起多轮研究核对模型输出的主线质量；真实新闻流核对去重的漏合/错合率（错合 = 0 是硬标准，漏合可容忍）。

### M-4 规格

manifest + service worker（静态资源 cache-first，API network-only，版本化清缓存）+ 图标（陶土红 Echo 印章）+ 设置页安装引导一行 + 通知点开深链 `#/watch/:ticker`。诚实记录平台限制：iOS 16.4+ 才支持已安装 PWA 的 Web Push，Telegram 通道仍是 iOS 的主通知路。

### 顺序理由

1. **M-1 打头是用户定调**，且 E9 快照是时间序列——每晚一天就少一天数据，越早上线积累越早开始。
2. **M-2 本应作为 bug 修复打头**，但与 M-1 零依赖（持仓页是全新 UI，直接按 §3.5 移动优先建，不经过坏掉的 wl-row 布局），顺位第二不吃亏。
3. **M-3 让看盘的差异化真正成立**——否则 M-2 修好的是一个内容空心的页面。
4. **M-4 收尾**，把前三者装进手机桌面。全轨 ≈1.5-2 周，不动后端研究链路，风险低。

**之后的方向（本轨不做，顺序不许倒）**：U 轨（部署 + auth + 多用户）→ 增长 → 商业化（P8 维持缓做）。

## 3.5 体验与设计标准（用户定调，全轨适用，每阶段"完成"定义的一部分）

- **品牌语言承接**（`07-brand.css`）：牛皮纸底、墨色字、陶土红点睛——"研究札记，不是交易终端"。新页面不另起颜色语义：涨跌色沿用现有（升绿 `#1c8c4a` / 跌用 `--danger`），状态色沿用（已证伪/有风险/逻辑还在）。
- **动效原则（calm motion）**：每个动效必须有目的（引导视线/确认操作/表达连续性），时长 150–350ms，缓动统一用现有 foundation token；入场动画一次性不循环；`prefers-reduced-motion` 下全部降级为即时呈现。**不引入动画/图表/组件依赖**（红线 14）——CSS transition/keyframes + 手写 SVG 覆盖全部需求。
- **交互质量门槛**：点击反馈 <100ms（乐观更新沿用 watch 页的既有模式：先本地变更立即渲染，后台对账）；加载态一律 skeleton，不白屏不闪跳；数字列 `tabular-nums` 定宽防布局跳动。
- **视觉验收（硬门槛）**：每个 M 阶段合并前，375 / 768 / 1280 三档视口 × light/dark 双主题**实跑截图**；无横向溢出；触控目标 ≥40px。

## 4. 明确不做（防后人走弯路，承接 v1–v3 全部红线）

1. **不换前端框架**：vanilla ESM + 事件委托。
2. **不做完全自主 ReAct 循环**：规划步数 ≤3，规则优先。
3. **不做 A 股**：聚焦 HK + US。
4. **不给买卖指令**：合规红线，宪法约束。
5. **不编数字**：取不到就"未核到"，禁止估计范围/反推。
6. **不在没有授权前把腾讯港股行情当商用数据源**（商业化阻断项）。
7. **不做过早抽象**：EA-6 等真实用户；R6 指引结构化等真实使用反馈；E7 名单可配置化等注册表需求。
8. **canary 不进 CI**：CI 保持无 key、不烧配额。
9. **涉及真实用户数据的 schema change 必须走非破坏性 migration**。
10. **factGuard 升档必须以落库的真实误报率为依据**：每一档都要真实样本支撑。
11. **近似口径必须显式标注**：历史 PE 分位、港股股本趋势等，不许把近似当精确。
12. **股东回报评分只认交易所一手**：HKEX / EDGAR / 公司公告，新闻转述不作数。
13. **业绩后复核不做"荐股化"表达**：beat/miss 只陈述事实与证伪线状态。
14. **（新增）动效必须克制且可关闭**：`prefers-reduced-motion` 全量降级；不引入前端动画/图表/组件依赖——净值曲线继续手写 SVG（红线 1 的延伸）。
15. **（新增）组合净值不造历史**：快照从任务上线日开始，缺日就是断口，不插值、不回填、不外推；跨币种折算口径页面显式标注（红线 5/11 的延伸）。
16. **（新增）UI 改动必须三档视口 × 双主题实跑验收**：375/768/1280 × light/dark 截图；移动端布局回归是硬检查——C1 修过一次横向溢出，P10 又坏在同一个页面家族，这类问题只能靠实跑抓（§3.5 视觉验收的落地）。

## 6. 怎么跑 + "完成"的定义（速查，承接 v1–v3）

```bash
npm install                 # 只有 better-sqlite3 一个原生依赖
npm run seed                # 建/重置本地 SQLite 种子库
npm run dev                 # http://127.0.0.1:4173
npm test                    # 全量用例必须全绿（EXIT=0）再提交
npm run lint                # eslint（correctness 级）
npm run typecheck           # tsc 全量 0 error（已进 CI，勿回潮）
npm run doctor              # 能力体检；--live 才发网络探活
npm run canary              # 真实数据 canary（本机/scheduler 职责，不进 CI）
# 后端无热重载！改 src/server/** 或 src/*.js 后要重启 node
# 隔离测试：LUVIO_DB_PATH=$TMPDIR/x.db PORT=4199 node server.js
```

**一次改动的"完成"= 代码 + 对应测试（进 `tests/`，接入 `npm test`）+ §5 状态表标 ✅ + 一条中文 commit。** 浏览器可见的改动必须实跑验证；**本轨额外要求：UI 改动的"完成"包含 §3.5 视觉验收（375/768/1280 × light/dark 实跑截图、无横向溢出、动效含 reduced-motion 降级）**。涉及外部数据源的改动必须真实调用验证。代码改动走 branch + PR 进 main；纯文档改动可直接 commit main。

**key 都在 `.env`（gitignored）**：GLM / DEEPSEEK / FMP / FINNHUB / TWELVEDATA / TAVILY。沙箱会墙搜索引擎，端到端 web 证据效果要在用户本机验证。
