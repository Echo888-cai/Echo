## 2. Gap 清单（三视角，每条含七要素）

> 字段约定：**问题 / 为什么重要 / 用户价值 / 工程风险 / 建议优先级 / 需要真实数据验证? / 预计工作量**。
> 优先级：P1 = 建议下一程就做；P2 = 第二程候选；P3 = 记录在案等时机；P4 = 明确缓做。

### 视角一 · 专业投资研究员（R：距离真正的 research desk 差什么）

**R1 · 财报日历缺位——系统不知道"下一份财报什么时候来"** ★ ✅ 已完成（G-2，2026-07-05）
- **问题**：全链路没有任何 earnings calendar 数据源。决策面板的 action 高频输出"等财报/公告验证"（`decisionPanel.js` `composeAction`），scheduler 有盘前速报/证伪巡检，但没有任何组件知道下一个业绩日在哪天。
- **为什么重要**：研究桌的工作节奏是围绕业绩日组织的——业绩日前提醒"你的证伪条件即将被检验"，业绩日后触发"指引 vs 实际"复核。没有日历，"等财报验证"是一句永远没有下文的话。
- **用户价值**：高。直接强化已有闭环（研究→证伪条件→监控→提醒），看盘卡可显示"下一业绩日 T-5"。
- **工程风险**：低。Finnhub `/calendar/earnings` 免费档覆盖美股；港股无免费日历，可从 HKEX 公告节奏近似或诚实标"未核到"（fail-safe 原则）。
- **优先级**：P1 ｜ **真实数据验证**：需要（Finnhub 免费档字段/配额、港股可得性实测）｜ **工作量**：≈1 天。

**R2 · 估值缺同业与历史锚——宪法要求"与历史区间和同业匹配"，引擎给不出** ★ ✅ 已完成（G-3，2026-07-05；历史分位仍未做，见下）
- **问题**：PE 法用"当前 PE ±30%"当 bear/bull（`valuationEngine.js:306-311`），本质自参照；EV/Sales 用增速分档的**规则默认倍数**（`:185-187`），不看真实同业。宪法估值纪律（`prompts.js` 第五条）要求说明"与历史区间和同业是否匹配"，但引擎没有任何同业倍数或历史分位数据——这段纪律只能靠模型脑补，恰是事实块红线要防的行为。
- **为什么重要**：估值条/赔率是产品核心卖点，"±30% 机械带"在专业用户眼里一眼假；同业锚也是"贵不贵"唯一有说服力的语境。
- **用户价值**：高。估值带从"机械带"变成"有对照的带"，散文与数字终于同源。
- **工程风险**：中。同业对标可落地（`SECTOR_MAP`/curated 名单/本地 companies 池选 2-3 家同业拉 PE/EV-S 取中位，对比管道 `buildCompareSummary` 现成）；**自身历史估值分位免费档拿不到**（需历史 PE 序列）——先做同业、历史分位诚实标"未核到"。
- **优先级**：P1 ｜ **真实数据验证**：需要（同业倍数免费档覆盖度实测）｜ **工作量**：≈2 天。

**R3 · 事实块红线只是提示词，没有数字级代码校验** ✅ 已完成（2026-07-05，见 §5 状态跟踪表）
- **问题**：宪法禁止模型给出事实块之外的财务数字，但没有任何代码校验模型散文里的数字是否真在已接地数据里。B-1 的 `reconcileConfidence` 只在"面板置信度"粒度工作，防不了正文里编一个"收入约 2-3 亿"。
- **为什么重要**：防幻觉是产品第一信任支柱；提示词纪律会随模型版本/回答长度漂移，只有代码校验可靠——与 B-7/D1 的经验同构。
- **用户价值**：高但不可见（价值在避免一次信任崩塌）。
- **工程风险**：中高。数字抽取/匹配的**误报**会把合法推演误标（汇率折算、单位换算、"约十年"这类非财务数），必须用真实模型输出反复调阈值，宁可漏报不可误报。
- **优先级**：P2（先有 G-1 稳定的事实块口径与观测底座再做）｜ **真实数据验证**：必须（真实研究回答标注）｜ **工作量**：≈2-3 天。

**R4 · 对比缺"自动可比公司"** ✅ 已完成（随 G-3 一并做掉：Finnhub 同业清单已接入估值引擎与前端"同业对照"面板）
- **问题**：对比要用户自己点名两边（compareWith / EA-2 比较句式）；研究员会主动给同业对照。单公司研究报告里没有"同业 2-3 家关键倍数对照"块。
- **为什么重要**：没有对照系，"贵不贵"没有语境；与 R2 同源同数据。
- **用户价值**：中高 ｜ **工程风险**：低中（行业结构已有；FMP peers 端点免费档需实测）。
- **优先级**：P2（与 R2 同期做可摊薄成本）｜ **真实数据验证**：需要 ｜ **工作量**：≈1 天（若与 R2 同期）。

**R5 · 组合分析停留在"体检"，缺研究联动**
- **问题**：`portfolioReview` 有集中度/止损/触线/回撤，但没有行业/市场/币种暴露，也不联动研究结论——比如"你 2 笔持仓的证伪线距现价 <5%""3 笔持仓集中在同一赛道"。持仓页只是列表 + 体检卡。
- **为什么重要**：研究→持仓是产品闭环的收口；用户点名的候选方向之一。
- **用户价值**：中高（对有持仓的用户是高）｜ **工程风险**：低（全部本地数据派生，无新数据源）。
- **优先级**：P2 ｜ **真实数据验证**：不需要 ｜ **工作量**：≈2 天。

**R6 · 管理层指引没有结构化沉淀**
- **问题**：8-K 抽取（美股）与 HKEX 解析拿到的是"事件/三表"，但**指引数字**（下季收入/毛利率指引）没有独立落库；证伪条件锚指引数字靠模型当轮记忆，跨会话即丢，"指引 vs 实际"无法自动核对。
- **为什么重要**：guidance vs actual 是研究员判断"预期差"的主工具；宪法要求"当前倍数对应哪一年预期"。
- **用户价值**：高但兑现慢 ｜ **工程风险**：高（抽取准确率/口径统一/期次对齐都难）。
- **优先级**：P3（先用 R1 把业绩节奏立起来，指引结构化等真实使用反馈再上——防过早抽象）｜ **真实数据验证**：必须 ｜ **工作量**：≈3+ 天。

### 视角二 · 产品体验（P：哪里还不像一个真正的研究终端）

**P1 · 数据源健康对用户（和开发者）不可见** ★ ✅ 已完成（G-1，2026-07-05；G-2/G-3 的 earnings/comp_peers 探测已并入同一面板）
- **问题**：接地条只说"这一轮新闻✗"，不说"新闻源这周一直挂"。`doctor` 是 CLI 且默认不发网络请求；设置页只有 scheduler 状态。当前研究的港股有没有一手 HKEX 数据、上次后台刷新是否失败，全都无处可看。
- **为什么重要**：免费源体系天然脆弱（FMP 402、Yahoo 403、腾讯魔法字段位），**不可见的降级会被读成"产品不行"；可见的降级反而是诚实卖点**——与"诚实置信度"的品牌承诺一致。
- **用户价值**：高 ｜ **工程风险**：低（读已有 providerStatus/E1 canary 结果，纯展示层）。
- **优先级**：P1（与 E1/E3 同一阶段交付）｜ **真实数据验证**：需要（面板数据本身来自真实探测）｜ **工作量**：与 E1/E3 合计 ≈2-3 天。

**P2 · 证伪通知回链停在 ticker 级**
- **问题**：`watch_rules` 落库时记了 `session_id`（当时定下证伪条件的那轮研究），但通知点开只带 `data-ticker`（`notifications.js:99`）——用户收到"证伪命中"后回不到"当时为什么定这条线"的上下文。
- **为什么重要**：证伪闭环的最后一步是"复核当时的逻辑"；回链断了，闭环就只剩提醒。
- **用户价值**：中高 ｜ **工程风险**：低（数据都在，只差前端跳转路径）。
- **优先级**：P2 ｜ **真实数据验证**：不需要 ｜ **工作量**：≈0.5 天。

**P3 · 研究历史没有全文检索**
- **问题**：侧栏按对话分组列历史，但没有跨公司/跨会话搜索。"我上次对 X 的判断"单公司有画像 Tab 兜着，跨公司问题（"我研究过哪些液冷相关的票"）没有入口。研究沉淀越多此缺口越痛。
- **优先级**：P3（SQLite FTS5 低成本可落地）｜ **真实数据验证**：不需要 ｜ **工作量**：≈1 天。

**P4 · 移动端 C1 余项 + C2 PWA**
- **现状**：阻断性横向溢出 bug 已修（2026-07-05）；剩触控区加大、侧栏抽屉化、composer 固定底、manifest + service worker。用户已明确保持 🟡 顺手策略、不急。
- **优先级**：P3 ｜ **真实数据验证**：需要真机/多档宽度实测 ｜ **工作量**：≈2 天。

**P5 · Onboarding / 首次体验**
- **现状盘点**：其实不差——研究页空状态有品牌 hero + 4 张示例卡 + 能力 pill（`research.js:775-800`），看盘/持仓都有空状态引导 CTA。真正缺的是"第一次看到决策面板"时的术语解释（评级体系/赔率/证伪是什么）。
- **为什么缓做**：当前唯一用户是作者本人，onboarding 的 ROI 在接近有真实外部用户时才兑现。
- **优先级**：P4（明确缓做，防止误把它当低垂果实）｜ **工作量**：≈1-2 天（到时再估）。

## 6. 怎么跑 + "完成"的定义（速查，承接 v1）

```bash
npm install                 # 只有 better-sqlite3 一个原生依赖
npm run seed                # 建/重置本地 SQLite 种子库
npm run dev                 # http://127.0.0.1:4173
npm test                    # 415 用例，必须全绿（EXIT=0）再提交
npm run lint                # eslint（correctness 级）
npm run typecheck           # tsc 全量 0 error（D1 成果，现已进 CI，勿回潮）
npm run doctor              # 能力体检；--live 才发网络探活
npm run canary              # G-1：真实数据 canary（0700/9988/9868.HK + AAPL/NVDA 全管道探测，落库供设置页健康面板读）
npm run hk-coverage -- --limit=20   # G-1：港股一手 filing 覆盖率增量扫描（654 支，可反复跑；--rescan 重查失败原因）
# 后端无热重载！改 src/server/** 或 src/*.js 后要重启 node
# 隔离测试：LUVIO_DB_PATH=$TMPDIR/x.db PORT=4199 node server.js
```

**canary 不进 CI**：`npm run canary` / `npm run hk-coverage` 都是本机/scheduler 职责，会真实调用外部数据源、消耗配额、且需要 `.env` 里的 key——CI 环境没有这些，也不该有。

**一次改动的"完成"= 代码 + 对应测试（进 `tests/`，接入 `npm test`）+ §5 状态表标 ✅ + 一条中文 commit。** 浏览器可见的改动必须实跑验证；涉及外部数据源的改动必须真实调用验证（B-7 教训）。代码改动走 branch + PR 进 main；纯文档改动可直接 commit main。

**key 都在 `.env`（gitignored）**：GLM / DEEPSEEK / FMP / FINNHUB / TWELVEDATA / TAVILY。沙箱会墙搜索引擎，端到端 web 证据效果要在用户本机验证。

## 7. R7 研究记分卡 / 自动复盘（已对齐并完成，2026-07-06）

> **状态**：**已完成**。用户已对齐方向并明确要求 Phase A/B/C 一次性做完（不采用下面 Phase B 原建议的"等样本成熟再排期"）。实现细节与真实数据验证见 §5 状态跟踪表 R7 行；以下保留原始论证存档。

**问题**：产品从 v1 起就宣称"让这次研究成为下次复盘的可对比资产"（`src/prompts.js:301`，深度研究框架原文），`decisionPanel`/`company_profiles`/`profile_events` 也确实把每次判断、理由、证据链接都存了下来——但**没有任何代码把"当时说了什么"和"后来实际发生了什么"做对比**。`company_profiles.valuation_json` 只存最新一次的估值带，历史快照不保留；`profile_events` 记了判断变化的理由，但不记当时的估值带/价格。用户现在只能凭记忆判断"我上次说腾讯低估，说得对不对"，Echo 自己也从不检验自己给出的判断质量。

**为什么重要**：这是产品"投资纪律 + 诚实置信度 + 证伪条件"这套身份承诺缺的最后一环——证伪条件、估值带如果从不被事后核对，就只是研究时的仪式感，说服力打折。而且这恰好是**现在**才第一次具备条件做的方向：早前画像库是空的，没有历史快照可比；G-2/G-3/R3/估值叙事强化之后，估值带、同业锚、财报日历都已经稳定进入正文和结构化字段，"当时说了什么"第一次变得足够精确、可结构化留痕。

**用户价值**：高但间接——不会让下一次研究"更准"，但会让用户对系统整体可信度产生复利式信任（"这工具会为自己的判断负责，不是只会自圆其说"），长期还能暴露系统性偏差（比如是否总体偏乐观、港股判断准确率是否系统性低于美股）反哺后续改进。

**工程风险**：中，且风险分布不均匀——数据沉淀（Phase A）风险很低，展示/统计（Phase B）风险在于"样本太小时呈现会显得空洞甚至打脸"。

**数据需求**：**不需要新外部数据源**。历史收盘价已有可复用逻辑（`marketData.js` 算 `oneMonthPct`/`ytdPct` 用的历史 close 序列可直接复用去核"价格后来有没有落在估值带里"）；估值带/证伪条件/价格已经是研究流程里现成计算出的字段，缺的只是"按时间点快照下来"这一步。

**真实数据验证的特殊局限（提前说明，避免后续误判进度）**：实测过当前画像库——最老的画像（0700.HK）建档于 2026-07-04，其余全部是 2026-07-05 当天建的（`node -e` 查过 `company_profiles.created_at`，见此次对话记录）。这意味着：
- Phase A（快照落库）可以立刻做且能立刻验证"落库逻辑对不对"（构造历史数据回放）；
- 但"记分卡统计出的准确率有没有意义"这件事**现在没有条件验证**——样本量为零到个位数，任何百分比都是噪音。这不是工程缺陷，是时间没到；必须在 Phase B 的展示层显式做"样本不足，暂不显示统计口径"的诚实降级（呼应本项目一贯的 fail-safe 原则：宁可说"还不能算"也不假装有效）。

**已考虑但排除的替代方向**（说明为什么不是这次的选择，供后续参考）：
- **P8 商业化 / EA-6 自定义 skill**：仍然缺少触发信号（真实外部用户/真实自定义需求），维持 §3/§4 已有的"等真实需求"判断，本次没有新证据改变这个结论。
- **R6 指引结构化**：同样依赖"真实使用反馈"，且抽取准确率风险高于 R7（管理层指引数字的抽取远比价格历史读取脆弱）。

- 之所以选 R7：它是唯一一个"现在做"能拿到即时收益（哪怕只是 Phase A 打点）、又不需要等待真实用户/真实需求信号的方向——数据条件（画像库 + 价格历史）已经自然具备，只是此前没人把这两块连起来。
