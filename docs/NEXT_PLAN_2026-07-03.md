## 0. 现状（动手前先懂这个）

Echo 现在是"**单公司深度问答**"最强、"**跨标的发现**"刚起步的形态。两条链路是**分开的两个筒仓**：

| 链路 | 入口 | 服务 | 特点 |
|------|------|------|------|
| 公司研究 | `/api/chat` | `agentService.js` | 单 ticker，产出完整 decisionPanel（估值条/证伪/溯源/置信度） |
| 发现层（P6） | `/api/discover` | `discovery.js` | 筛选器 `runScreener` + 宏观 `runMacro`，不绑定公司 |

**分流发生在前端**：`src/ui/resolve.js:303` 的 `discoveryKindOf(question)` 在公司解析**之前**判断——命中筛选/宏观就打 `/api/discover`，否则走公司解析 → `/api/chat`（`src/ui/research.js:435`）。

**会话是单公司绑定**：`research_sessions` 表主键带 `ticker`（`src/db/index.js:98`，`FOREIGN KEY (ticker)`）。发现层结果只是"落回当前线程"，不是一等公民会话。

**已具备、可复用的地基**（别重造）：
- 筛选器 `runScreener`（`discovery.js:153`）：中文条件解析（行业/PE/价格/市值）→ FMP `company-screener` + 已研究画像池，返回名单，前端可"一键开研究"。
- 公司解析 `resolveCompany`（别名表 + FMP 名搜 + LLM 解析并回验）。
- 模型网关 `modelGateway`（多provider fallback）、`intentClassifier`、`webEvidenceService`。

**一句话**：脑和可视化很强，但它现在**被动等你点名一家公司**；用户要的是一个**会自己规划检索、能同时端上多家公司、能比较**的研究员。

## 1. 北极星：一个统一的、会用工具的 Agent

把"前端预判筒仓"升级成"**一个入口 + 服务端 Agent 规划 + 工具层**"。用户问什么都进同一个口，Agent 决定调用哪些能力、要不要串起来：

```
用户问题 ──► /api/ask（统一入口）
              │
              ▼  意图+规划（规则优先，必要时一步模型规划）
        ┌─────┴───────────────────────────────┐
        ▼         ▼          ▼          ▼        ▼
   resolveCompany screenStocks compareCos macroRead webEvidence
        │         │          │          │        │
        └──────► 组装回答（面板 / 名单 / 对比表 / 宏观短评）
                  每个公司结果 = 可开成一个"公司分会话"
```

**关键理念**：这不是推倒重来，是把已有函数包成"**工具**"，加一个**薄编排层**。先做**受控规划**（规则 + 至多一步模型出计划），不做完全自主 ReAct 循环——投研场景要的是可靠、可解释、可控成本，不是炫技的多跳自主。

## 2. 分阶段计划（EA = Echo Agent）

> 原则：每阶段可独立浏览器验证、进 `tests/`、单独提交。破坏性小、先地基后能力。

### EA-0 统一入口 + 服务端意图路由（地基，≈1 天）
- **为什么**：分流逻辑散在前端（`resolve.js:303` / `research.js:435`），做 Agent 前必须收敛到服务端一个地方。
- **改法**：新增 `src/server/routes/ask.js` → `/api/ask`，内部先跑 `intentClassifier` 统一判定 `company | screener | macro | compare`，再 dispatch 到现有服务。前端 `sendChat` 改为**只调 `/api/ask`**，服务端回 `{ kind, ... }` 决定渲染分支。旧 `/api/chat`、`/api/discover` 暂保留为内部实现，外部只暴露一个口。
- **验收**：同一输入框，"腾讯怎么样""筛美股半导体 PE<20""美股今晚有什么事"三类问题都从 `/api/ask` 正确分派；`tests/` 加意图分派单测；无回归。

### EA-1 工具层抽象（把已有能力包成可调用工具，≈1–2 天）
- **为什么**：Agent 要"会用工具"，先得有工具的统一签名。
- **改法**：新增 `src/server/services/agentTools.js`，把现有能力包成纯函数式工具，统一 `{ name, description, inputSchema, run() }`：
  - `resolveCompany(name)` → ticker（复用现有解析）
  - `screenStocks(filters)` → 名单（复用 `runScreener`）
  - `researchCompany(ticker, question)` → decisionPanel（复用 `agentService`）
  - `compareCompanies(tickers[])` → 对比数据（EA-3 落地）
  - `macroRead(question)` → 宏观短评（复用 `runMacro`）
  - `webEvidence(query)` → 证据（复用 `webEvidenceService`）
- **验收**：每个工具可单测（`tests/agentTools`），输入输出 schema 稳定；`ask.js` 改为调用工具而非直连服务。

### EA-2 受控规划器（让 Agent 会串工具，≈2 天）★核心
- **为什么**：这是"扩大 Agent 能力"的心脏。例：**"美股存储芯片有什么好标的"** 需要 `screenStocks(半导体/存储)` → 按盈利质量排序 → 端上名单；**"英伟达和 AMD 谁赔率好"** 需要 `resolve×2 → compareCompanies`。
- **改法**：`agentPlanner.js`——规则先行（能确定的直接给计划），不确定时让模型出**一个** JSON 计划（`[{tool, args}]`），校验后顺序执行、聚合结果。全程有超时与工具失败降级（任一工具挂了给"已取到的部分 + 诚实缺口"）。**红线**：计划步数上限（如 ≤3），防止自主循环烧钱/失控。
- **验收**：三类复合问题端到端跑通并浏览器可见；计划可在响应里回显（可解释）；模型不可用时规则兜底仍给结果。

### EA-3 跨标的能力补强（把"发现"做扎实，≈2 天）
- **为什么**：现在筛选器只认行业/PE/价格/市值，且**忽略**股息/增速（`discovery.js:72-73`）；"好标的"没有排序口径。
- **改法**：
  1. **子行业**：`SECTOR_MAP` 加"存储/内存"（Micron/SanDisk 等）、"设备""EDA"等细分（`discovery.js:24`）。
  2. **"好"= 可解释排序**：名单按现有**盈利质量分**（`financialQuality.js`）或估值分位排序，而非只按市值；"好标的"落地为"筛 + 排序"。
  3. **compareCompanies**：2–3 个 ticker 并排（估值/盈利质量/增速/共识），复用 `answerComposer` 的数据块；前端出对比表。
- **验收**："美股存储芯片龙头对比"给出并排表；"筛便宜的半导体"名单带盈利质量排序列；`tests/phase-ea` 覆盖新解析。

### EA-4 多公司对话 / 公司分会话（会话模型升级，≈2–3 天）
- **为什么**：用户明确要"以对话显示公司分会话"——一次研究里并列多家公司、可跳转，而不是每换一家就丢上下文或另起孤立会话。
- **改法**：
  1. **数据**：`research_sessions` 增 `conversation_id`（可空，默认自身），把"一次对话"作为分组；单公司仍是一行，多公司共享 `conversation_id`。迁移：老会话首跑 `conversation_id = id`（无损）。
  2. **前端**：发现层名单/对比结果里的公司，"打开"即在**当前对话**下新增一个公司分会话（tab 或分节），侧栏按对话分组、组内列公司；跳转不清空其他公司的已研究结果。
  3. **上下文**：切换公司分会话时，composer 的"当前标的"随之切换；追问归属正确公司（复用现有 `mentionsNewCompanyStrong`）。
- **验收**：一次对话里研究腾讯→筛半导体→打开 Micron，三者都在侧栏同一对话组下可来回跳；每个公司分会话保留自己的面板/估值条；刷新后结构不丢。

## 4. 风险与红线（沿用宪法，别踩）
- **不给买卖指令**：合规红线，`prompts.js` 宪法已约束；Agent 再强也只给研究判断/监控条件/风险检查点。
- **不编数字**：事实块红线（`prompts.js` "防幻觉"）对 Agent 同样生效；工具取不到就写"未核到"。
- **规划步数上限**：EA-2 计划 ≤3 步，防自主循环烧 token/失控。
- **港股行情源合规**：腾讯免费源商用有风险（ARCH-6），未获授权前不作商用数据源宣传。
- **不上大前端框架重写**：ARCH-1 既定；EA-4 用渐进方式加多会话，不引入 React/Vue 全家桶。

## 5. 每次怎么启动 + 校验（速查，教学版）

```bash
# 1. 启动（首次或换机）
npm install                     # 装依赖（只有 better-sqlite3）
npm run seed                    # 建/重置本地 SQLite 种子库
npm run dev                     # 起服务 → http://127.0.0.1:4173

# 2. 改完代码后的自检（每次提交前必跑）
npm test                        # 149 用例；必须全绿（EXIT=0）再提交
npm run lint                    # eslint 静态检查

# 3. 后端无热重载！改 src/server/** 或 src/*.js 后要重启 node：
#    Ctrl+C 停掉，再 npm run dev

# 4. 想在不污染本地库的临时库上跑：
LUVIO_DB_PATH=$TMPDIR/x.db PORT=4199 node server.js

# 5. 只跑某一套测试（定位问题时）：
npm run test:smoke              # 冒烟
npm run test:reliability        # 可靠性
node tests/phase6.mjs           # 发现层
```

**一次改动的"完成"定义**：代码 + 对应测试（进 `tests/`，接入 `npm test`）+ 本文件 §状态表标 ✅ + 一条中文 commit（说清做了什么、为什么）。**浏览器可见的改动**（UI/交互）要实跑截图验证，不许只靠"应该没问题"。
