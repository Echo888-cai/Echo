## 0. 接手协议（先读这条）

1. **分支**：`cleanup/pristine-refactor`。动手前 `git log --oneline -5` 确认没有比本文档更新的提交；有则先读那些提交的信息。
2. **跑起来**：`npm install && npm run seed && npm run dev` → http://127.0.0.1:4173。测试：`npm test`（必须全绿再开工，改完不许回退）。
3. **后端无热重载**：改 `src/server/**` 或 `src/*.js` 后必须重启 node 进程。本地隔离测试：`LUVIO_DB_PATH=$TMPDIR/x.db PORT=4199 node server.js`。
4. **key 都在 `.env`**（gitignored）：DEEPSEEK / FMP / FINNHUB / TWELVEDATA / TAVILY 已配。沙箱环境会墙搜索引擎，端到端效果要在用户本机验证。
5. **完成一项的定义**：代码 + 测试（进 `tests/`，接入 `npm test`）+ 本文档 §5 状态表标记 ✅ + 一条 commit（中文、说清做了什么和为什么）。
6. **用户的工作偏好**（来自长期协作）：PM 式评审、分阶段交付、每阶段实跑验证、"破而后立、先修地基"；不要口头承诺，要可复现的验收证据。
7. **不要做的事**：不要引入大框架重写前端（见 ARCH-1 的渐进方案）；不要在没有付费/官方授权前把腾讯港股行情当商用数据源宣传（ARCH-6）；不要给买卖指令（合规红线，提示词宪法已约束）。

## 2. 问题清单 — 交互与 UI（UX-x）

### UX-1 零主动性：所有价值藏在按钮后面 ｜ 严重度：高 ★核心
- **现状**：盘前事件 ◷ 是手动按钮（`/api/events/digest` 按需算），持仓触线提醒也只在点开时算。用户不打开网页，止损被击穿都不会知道。
- **为什么不合理**：投研助手的核心时刻是"开盘前 5 分钟"和"触线的那一刻"。被动形态下产品价值≈0×使用频率。HoneClaw 的 cron+推送是它的第一卖点。
- **改法**：见 §4 P1（scheduler + 通知中心 + Telegram）。
- **验收**：不开网页，盘前时点 Telegram 收到摘要；打开网页铃铛有未读。

### UX-2 研究记忆不可搜索 ｜ 严重度：中
- **现状**：侧栏历史只能滚动找（`src/app.js` `renderSessionHistory`）。
- **改法**：侧栏加搜索框，`/api/research/sessions?q=` 走 SQLite LIKE（title/ticker/question 三字段）。
- **验收**：搜"阿里"能过滤出所有相关会话。

### UX-3 无键盘效率层 ｜ 严重度：中
- **现状**：重度用户（目标付费人群=严肃个人投资者）没有任何快捷键；换标的要鼠标点。
- **改法**：`Cmd/Ctrl+K` 命令面板：输入代码/中文名直达研究或公司页；`/` 聚焦输入框。纯前端，复用 `resolveCompany`。
- **验收**：Cmd+K → "腾讯" → Enter 直接开研究。

### UX-4 移动端未适配 ｜ 严重度：中偏高
- **现状**：`src/styles.css` 是桌面栅格；看盘（最高频场景）在手机上不可用。
- **改法**：一轮响应式 pass：≤768px 侧栏折叠为抽屉、看盘列表全宽、composer 固定底部、字号/点按区加大。不追求完美，追求"手机上能看盘、能提问、能收通知"。
- **验收**：375px 宽度下三个 Tab 可用、无横向滚动。

### UX-5 降级不可见：用户不知道自己看到的是几手数据 ｜ 严重度：高
- **现状**：数据源静默降级设计得很好（fmpClient 冷却、多级 fallback），但**产品上不可见**——只有回答里的 grounding bar。没有一个"数据源健康"页面；无 key 首跑时用户不知道自己看的是本地兜底档案。
- **为什么不合理**：投研产品的命根是信任。"它是不是在拿真数据"必须一眼可查。
- **改法**：`/api/status` 已有 provider 状态 → 前端做"数据源"面板（设置区）：每个 provider 名称/用途/状态/冷却中原因；无任何模型 key 时首页顶部横幅提示"演示模式：回答基于本地档案"。
- **验收**：拔掉 FMP key 重启，面板显示 FMP 不可用及影响范围；无模型 key 时首页有演示模式横幅。

### UX-6 看盘整盘刷新慢 ｜ 严重度：中（上轮遗留）
- **现状**：看盘列表刷新逐标的串行等待，标的多时明显卡。
- **改法**：`watchDesk.js` 批量并发（限流 5-6 并发）+ 前端乐观渲染旧值+逐个到位更新。
- **验收**：10 个标的整盘刷新 < 3s（缓存冷启动）。

### UX-7 研究和监控是两个孤岛：证伪条件不落地 ｜ 严重度：高 ★闭环关键
- **现状**：研究会话产出高质量证伪条件（decisionPanel.falsification），但它们只活在那次回答里。看盘页/事件引擎完全不知道这些条件的存在。
- **为什么不合理**：这是"研究→纪律执行"的断环。用户研究时说"跌破 90 证伪"，两周后真跌破了，产品毫无反应——那研究的沉淀价值就是 0。HoneClaw 用画像+定向提醒做了这个闭环。
- **改法**：分两步。第一步（P2）：证伪条件随画像沉淀到公司维度，公司页显示"证伪监控"块；可量化的价格类条件（跌破 X/涨破 Y/PE>Z）解析成 `watch_rules` 表记录。第二步（P1 的 scheduler 就绪后）：盘前任务逐条核对 watch_rules，命中→通知。
- **验收**：研究里说"跌破 4.2 证伪"→ 公司页出现该监控项 → 价格真跌破后盘前摘要包含这条提醒。

### UX-8 空状态与新手引导薄弱 ｜ 严重度：低偏中
- **现状**：`renderEmptyState` 有示例问题（好），但没有讲清产品能做什么/不能做什么（HK+US、不给买卖指令）。
- **改法**：空状态加三行能力说明 + 一个"数据源状态"入口（配合 UX-5）。
- **验收**：新用户 30 秒内明白能问什么。

## 3. 问题清单 — 架构与工程（ARCH-x）

### ARCH-1 前端 2705 行单文件 ｜ 严重度：高 ★工程债之首
- **现状**：`src/app.js` 一个文件装下状态、API、路由、三个页面、所有组件渲染（2705 行）。已是 `type="module"` + `data-action` 事件委托（无内联 onclick），这是好基础。
- **为什么不合理**：改任何 UI 都要面对全文件；多人/多 AI 会话并行必然冲突；新功能边际成本持续上升。
- **改法（渐进，禁止大爆炸重写）**：按页面拆 ES modules，**零行为变更**：
  `src/ui/state.js`（状态+store）、`src/ui/api.js`、`src/ui/format.js`（esc/fmt*）、`src/ui/markdown.js`、`src/ui/research.js`、`src/ui/watch.js`（看盘+公司页）、`src/ui/portfolio.js`、`src/ui/components.js`（估值条/溯源卡等共享渲染）、`app.js` 只留路由+事件委托+初始化（目标 <400 行）。浏览器原生 ESM，不引入打包器。
- **验收**：`npm test` 全绿 + 手工过一遍三个 Tab 全部交互（研究/追问/对比/看盘增删/公司页/持仓/画像/导出）+ `app.js` < 400 行。

### ARCH-2 零 CI 门禁 ｜ 严重度：高（对"可交接"尤其致命）
- **现状**：`npm test` 覆盖不错但没人强制跑；push 到 GitHub 无任何检查。
- **改法**：`.github/workflows/ci.yml`：push/PR 触发，Node 22，`npm ci + npm test`（测试已能无 key 跑，模型层有本地兜底）。
- **验收**：GitHub Actions 页面绿勾。

### ARCH-3 无 lint ｜ 严重度：中
- **改法**：ESLint v9 flat config（`eslint.config.js`），只开 correctness 级规则（no-undef/no-unused-vars 等），不吵风格。`npm run lint` 进 CI。
- **验收**：`npm run lint` 0 error。

### ARCH-4 无 LLM 审计 ｜ 严重度：中偏高（商业化成本盲区）
- **现状**：`modelGateway` 调用不留痕：不知道每天烧多少 token、哪个 provider 失败率高、平均延迟多少。
- **改法**：`llm_audit` 表（ts/provider/model/purpose/prompt_chars/completion_chars/latency_ms/status/error），`modelGateway.callModel` 出入口埋点；`/api/status` 带最近 24h 汇总；将来接真实计费单价。
- **验收**：跑一轮研究后 `sqlite3 luvio.db 'select count(*) from llm_audit'` > 0，status 页能看到调用数/延迟。

### ARCH-5 单用户数据模型 ｜ 严重度：中（记录在案，暂不动）
- **现状**：portfolio/watchlist/sessions/profiles 都没有 user 维度。
- **决策**：**现在不做大迁移**（过早抽象）。约束：今后新表一律带 `user_id TEXT DEFAULT 'local'` 列；到 P8 商业化阶段做一次统一迁移（参考 HoneClaw 的 ActorIdentity 归一 + local→PG 双模式路线，它的 `memory/` crate 是现成教材）。
- **验收**：本文档存在此条 + 新表遵守约定。

### ARCH-6 港股行情依赖腾讯非官方接口 ｜ 严重度：高（商业化合规阻断项）
- **现状**：HK 实时价走腾讯财经免费接口（个人研究可用，商用无授权）。
- **决策**：个人使用阶段保留；**商业化前必须替换**为有授权源（候选：iTick / 富途 OpenAPI / LSEG / 恒生聚源；或 HKEX 直接授权）。P8 处理，本条防止遗忘。
- **验收**：商业化 checklist 含此项（见 P8）。

### ARCH-7 无部署形态 ｜ 严重度：中
- **现状**：只能本机 `npm run dev` + macOS 启动器；没有 Dockerfile、没有进程守护、没有数据备份。
- **改法**：`Dockerfile`（node:22-slim，volume 挂 `luvio.db` 与 `.env`）+ `docs/DEPLOY.md`（含 SQLite 备份一条命令）。scheduler（P1）就绪后，容器常驻即获得自动推送。
- **验收**：`docker build && docker run` 后功能与本机一致。

### ARCH-8 无环境自检 ｜ 严重度：低偏中
- **改法**：`npm run doctor`（`scripts/doctor.js`）：逐 key 检查存在性+连通性（打一个最便宜的探活请求），输出哪些能力可用/降级。对接手者和新用户都是第一入口。
- **验收**：故意删一个 key，doctor 能准确指出受影响能力。

### ARCH-9 无 schema 迁移机制 ｜ 严重度：中
- **现状**：`src/db/index.js` 建表靠 `CREATE TABLE IF NOT EXISTS` + 手工 ALTER，无版本号。
- **改法**：轻量迁移器：`user_version` PRAGMA + `src/db/migrations/NNN_*.sql` 顺序执行。不引依赖。
- **验收**：从旧库文件启动自动迁到最新版本，`npm test` 全绿。

## 4. 问题清单 — 功能（FEAT-x）与分阶段执行计划

> 按投入产出排序。每个 Phase 独立可交付、可验收；**做完一个 Phase 提交一次**。

### P0 工程门禁（≈半天）→ ARCH-2/3/8
CI + lint + doctor。先修地基：让之后每一步都有护栏。

### P1 主动性跳变：scheduler + 通知中心 + Telegram（≈1-2 天）→ UX-1/UX-8、FEAT-7 ★本计划最大产品跳变
- **后端**：
  - `src/server/services/scheduler.js`：进程内每分钟 tick；任务两种形态——每日定点（"HH:MM" 北京时间，如盘前 09:00 港股/21:15 美股）与间隔型（如每 30 分钟盘中触线检查）。任务登记表 `scheduled_jobs`（含 last_run_at），启动时补跑 misfire（进程不常驻的现实下，"打开即补跑"保底）。
  - `src/server/services/notifier.js`：`notify({kind,title,body,payload})` → 写 `notifications` 表 + 有 `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` 时推 Telegram（裸 fetch Bot API，不加依赖）。
  - 内置任务①盘前摘要：对"有持仓或有画像的公司"跑 `eventEngine` digest，高/中级事件 → 通知；②触线检查：盘中每 30 分钟核对止损/止盈/回撤（复用 eventEngine 的 position_alert）。
  - 路由：`GET /api/notifications?unread=1`、`POST /api/notifications/read`、`POST /api/notify/test`。
- **前端**：顶栏铃铛 + 未读红点（60s 轮询 unread count）；下拉通知面板（点击跳对应公司页/研究）；设置区 Telegram 配置说明。
- **验收**：①不开网页，到点 Telegram 收到盘前摘要（用户本机验证）；②Web 铃铛未读/已读闭环；③无 token 时 Web 通知照常；④`npm test` 新增 scheduler/notifier 用例全绿。

### P2 研究→监控闭环 + 看盘完成度（≈1 天）→ UX-6/UX-7、看盘遗留
- 证伪条件沉淀：research 完成时把 `panel.falsification` 写入公司画像；可量化价格条件解析进 `watch_rules`（ticker/kind/threshold/source_session）；公司页"证伪监控"块；盘前任务核对命中→通知。
- 看盘：整盘并发刷新（限流+进度）、列表排序（涨跌幅/名称/市场）与筛选（只看港/美/触警）。
- **验收**：见 UX-6/UX-7 各自验收。

### P3 组合体检（≈1 天）→ FEAT-3（上上轮 P1 遗留）
- `/api/portfolio/review`：总市值/币种拆分、单一持仓集中度（>30% 预警）、市场暴露（HK/US）、距止损止盈分布、无止损持仓点名（纪律漏洞）、组合级一句话判断（走模型，注入真实组合数据）。
- 前端：持仓面板顶部"组合体检"卡。
- **验收**：3 笔测试持仓能产出集中度预警与无止损点名；测试进 `npm test`。

### P4 公司画像文档化（≈1-2 天）→ FEAT-6、UX-7 深化（对标 HoneClaw 最值得抄的一件事）
- 画像从"SQLite 字段投影"升级为"主档案 + 事件时间线"：`profile` 保留结构化字段之外新增 markdown 主体（投资主线/关键指标/风险台账/证伪条件演进）；`profile_events` 表按日期追加"判断变化"记录（为什么变、证据链接）。
- 前端：公司页新增"画像" Tab：主档案渲染 + 时间线 + 导出 Markdown。
- **原则**（抄 HoneClaw invariants）：画像是长期研究资产，不是交易日志；只有判断变化才记事件；保留 rationale 使未来复盘可行。
- **验收**：对同一公司三轮不同结论的研究后，时间线出现≥2 条变化记录且各带理由；导出的 Markdown 可读。

### P5 前端模块化（≈1-2 天，纯重构）→ ARCH-1
- 按 ARCH-1 改法执行。放在 P4 之后是刻意的：先让功能到位，再做无行为变更的拆分（拆分期间冻结其他前端改动）。
- **验收**：见 ARCH-1。

### P6 发现层：筛选器 + 宏观路由（≈1-2 天）→ FEAT-2/FEAT-4
- 筛选：意图路由识别"帮我筛…"（如"美股半导体 PE<20"）→ FMP screener API（免费档可用部分）+ 本地已研究公司池 → 结果表格 + 一键开研究。
- 宏观：intentClassifier 加 `macro` 意图（"美股今晚怎么看/美联储议息"）→ 不绑定公司，走指数行情（SPX/NDX/HSI）+ 网页证据 + 宏观提示词模板。
- **验收**："美股今晚有什么关键事件"不再被当成公司问题；筛选能返回真实名单并可点开研究。

### P7 港股一手数据管道（≈3-5 天，最大数据工程）→ FEAT-1/FEAT-5（B-P2 遗留）

## 6. 本次评审明确不做/否决的方向（防止后人走弯路）

1. **不换前端框架**（React/Solid 重写）：现有 vanilla ESM + 事件委托 + 服务端无构建的组合，对单人+AI 协作维护成本最低。拆模块（P5）就够。
2. **不自研全功能 cron 表达式**：两种任务形态（每日定点/间隔）覆盖全部真实需求。
3. **不做 A 股**：数据源、合规、竞争格局完全不同，聚焦 HK+US。
4. **不追 HoneClaw 的多渠道矩阵**（Discord/飞书/iMessage）：先 Telegram 单渠道打穿"主动性"价值验证，其余渠道等有真实用户需求。
5. **不在回答里引入图表图片生成**（matplotlib 路线）：前端 SVG 组件（估值条/价格曲线）信息密度已更高，纯文本渠道（Telegram）用文字摘要即可。
